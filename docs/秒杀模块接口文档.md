# 秒杀模块接口文档

## 1. 文档说明

- 文档范围：基于当前仓库代码整理，覆盖秒杀模块的管理端、用户端、首页端、ES 运维接口。
- 代码基准：
  - `backstage-system/src/main/java/com/backstage/system/controller/seckill`
  - `backstage-system/src/main/java/com/backstage/system/controller/homepage/OshHomePageSeckillController.java`
  - `backstage-system/src/main/java/com/backstage/system/service/impl/seckill`
- 整理日期：2026-06-02
- 说明：本文档以“当前代码真实行为”为准，包含部分实现细节和注意事项，不仅是路由清单。

## 2. 模块总览

秒杀模块当前对外接口分为 5 组：

1. 管理端商品池接口：`/pc/seckill/goods`
2. 管理端活动接口：`/pc/seckill/activity`
3. 用户端秒杀接口：`/pc/seckill/user`
4. 首页热门秒杀接口：`/pc/homepage/seckill`
5. ES 索引运维接口：`/pc/seckill/es`

## 3. 基础约定

### 3.1 认证方式

#### 3.1.1 管理端接口

- 使用 Spring Security 权限校验。
- 典型权限如下：
  - `seckill:goods:list`
  - `seckill:goods:add`
  - `seckill:activity:list`
  - `seckill:activity:add`
  - `seckill:order:list`

#### 3.1.2 用户端登录接口

- 需要在请求头传 `token`。
- 认证过滤器读取请求头：`token`
- 无 token 或 token 失效时，接口返回：

```json
{
  "code": 401,
  "msg": "未登录或Token已过期"
}
```

#### 3.1.3 匿名接口

带 `@Anonymous` 的接口可直接访问，不强制要求登录。当前主要包括：

- `GET /pc/seckill/user/activity/list`
- `GET /pc/seckill/user/activity/detail/{id}`
- `GET /pc/seckill/user/recent/orders`
- `GET /pc/seckill/user/announcement/notices`
- `POST /pc/seckill/user/esSync/all`
- `GET /pc/homepage/seckill/hot`
- `POST /pc/seckill/es/recreateIndex`
- `POST /pc/seckill/es/syncAll`

### 3.2 请求头

| Header | 必填 | 说明 |
| --- | --- | --- |
| `token` | 登录接口必填 | 用户登录态令牌 |
| `Content-Type: application/json` | Body 为 JSON 时必填 | POST/DELETE 带 JSON 时使用 |

### 3.3 公共响应格式

#### 3.3.1 `R<T>` 单对象/列表响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | Integer | 200 成功，500 业务失败，401 未登录 |
| `msg` | String | 响应消息 |
| `data` | Object/Array/null | 实际返回数据 |

#### 3.3.2 `TableDataInfo` 分页响应

```json
{
  "total": 23,
  "rows": [],
  "code": 200,
  "msg": "查询成功"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `total` | Long | 总记录数 |
| `rows` | Array | 当前页数据 |

### 3.4 分页参数

所有走 `TableDataInfo` 的列表接口，默认支持以下查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `pageNum` | Integer | 否 | 页码，默认 1 |
| `pageSize` | Integer | 否 | 每页条数，默认 10 |
| `orderByColumn` | String | 否 | 排序字段 |

- 用户端活动列表在 ES 查询路径下，内部按 ES 规则排序，`orderByColumn` / `isAsc` 不一定生效。
- 用户端活动列表在 MySQL 降级路径下，使用 `startPage()` 分页。

## 5. 核心业务规则

### 5.1 `seckillNo` 和 `orderNo` 的区别

这是秒杀模块最重要的概念之一：

| 字段 | 含义 | 生成时机 | 用途 |
| --- | --- | --- | --- |
| `seckillNo` | 秒杀尝试号 | Lua 扣库存成功后立即生成 | 轮询秒杀结果、幂等控制、取消秒杀单 |
| `orderNo` | 统一订单号 | 异步消费者建单成功后生成 | 支付、查支付状态、调支付系统取消 |

结论：

- 前端发起秒杀后，第一时间拿到的是 `seckillNo`，不是 `orderNo`
- 前端必须继续轮询结果接口，等 `orderNo` 返回后再进入支付流程

### 5.2 用户端“进行中活动”的判断规则

用户端活动列表和详情并不单纯依赖 `status = 2`，而是按“时间窗口”判断：

1. 活动不能是下架状态 `status = 4`
2. `startTime <= 当前时间`
3. `endTime >= 当前时间`

因此即使定时任务尚未把状态更新为 `2`，只要时间已进入活动区间，用户端逻辑仍可能判定活动可见、可抢。

### 5.3 秒杀执行的并发控制

`POST /pc/seckill/user/do/{activityId}/{itemId}` 的关键控制点：

1. 先校验活动时间、状态、商品归属
2. Redis 检查用户是否已有未完成秒杀流程 `seckill:order:{activityId}:{itemId}:{userId}`
3. 执行 Lua 脚本原子完成：
   - 扣减库存
   - 校验每人限购
   - 增加用户已购占用数
4. 成功后写入 `seckillNo`
5. 发送 Kafka 异步创建统一订单

### 5.4 秒杀接口限流

执行秒杀接口带有限流注解：

- Key 前缀：`seckill:doSeckill:`
- 维度：IP
- 时间窗：5 秒
- 次数：3 次

即同一 IP 在 5 秒内最多触发 3 次该接口。

### 5.5 用户端活动列表 ES 降级规则

`GET /pc/seckill/user/activity/list` 默认优先走 ES 搜索；ES 不可用时降级到 MySQL。

注意：

1. ES 路径支持：
   - `title`
   - `goodsType`
   - `tagNameList`
2. MySQL 降级路径当前只支持：
   - `title`
   - `goodsType`
3. 也就是说：当 ES 降级到 MySQL 时，`tagNameList` 会失效

### 5.6 活动发布与缓存预热

活动从草稿发布为未开始 `status = 1` 时，会提前预热 Redis：

- `seckill:activity:{activityId}`
- `seckill:item:{itemId}`
- `seckill:stock:{activityId}:{itemId}`

活动下架时会清理上述缓存及已购集合。

### 5.7 活动修改规则

#### 5.7.1 修改活动主记录

- 只有草稿活动 `status = 0` 允许修改

#### 5.7.2 修改活动明细 `items`

真实代码行为如下：

- `items = null`：不修改明细
- `items = []`：当前实现也不会修改明细
- `items` 非空：视为全量替换
  - 旧明细逻辑删除
  - 新明细重新插入

### 5.8 商品标签更新规则

商品池更新接口中：

- `tagNames = null`：不修改标签
- `tagNames = []`：清空标签
- `tagNames = ["Java", "后端"]`：全量替换为新标签

## 7. 管理端商品池接口

### 7.1 商品预览

- 请求方式：`GET`
- 路径：`/pc/seckill/goods/preview`

- 作用：根据 `goodsType + goodsId` 预览商品基础信息，供前端自动回填表单

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `goodsType` | Integer | 是 | 商品类型，当前仅支持 `1-课程`、`2-书籍` |
| `goodsId` | Long | 是 | 原始商品 ID |

#### 请求示例

```http
GET /pc/seckill/goods/preview?goodsType=1&goodsId=101
```

#### 成功响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
```

- `goodsType` 不支持：`暂不支持的商品类型：3`

### 7.4 添加商品到秒杀商品池

- 请求方式：`POST`
- 路径：`/pc/seckill/goods/add`
- 权限：`seckill:goods:add`
- Body：JSON

#### 请求体

```json
{
  "goodsId": 101,
  "goodsType": 1,
  "minSeckillPrice": 9.90,
  "sort": 100
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `goodsId` | Long | 是 | 原始商品 ID |
| `goodsType` | Integer | 是 | 商品类型，当前仅支持 1/2 |
| `minSeckillPrice` | BigDecimal | 是 | 最低秒杀价，必须大于 0 |
| `sort` | Integer | 否 | 排序权重，默认 0 |

#### 真实业务行为

1. 后端会自动读取原始商品信息，不需要前端传：
   - `goodsName`
   - `goodsCover`
   - `originPrice`
2. 同一 `goodsId + goodsType` 不允许重复入池
3. 新建后商品池状态固定为 `0-待审核`
4. 会自动从课程/书籍标签表同步标签到秒杀标签表

#### 常见失败场景

- `该商品已在秒杀商品池中，请勿重复添加`
- `课程不存在，ID：xxx`
- `电子书不存在，ID：xxx`

差异：

- 该接口挂在用户端 controller 下
- 返回消息更简单：`R.ok(count, "ok")`

建议前后端与运维统一使用 `/pc/seckill/es/syncAll`，语义更清晰。

## 12. 秒杀状态流转

### 12.1 活动状态流转

```text
草稿(0)
  └─ 手工发布 ─> 未开始(1)
        └─ 到达开始时间（定时任务） ─> 进行中(2)
              └─ 到达结束时间（定时任务） ─> 已结束(3)

未开始(1) 或 进行中(2)
  └─ 手工下架 ─> 已下架(4)
```

### 12.2 秒杀订单状态流转

```text
执行秒杀成功
  └─ 返回 status=-1（处理中，接口态）
        └─ 异步建单完成 ─> 待支付(0)
              ├─ 支付成功 ─> 已支付(1)
              ├─ 用户取消 ─> 已取消(2)
              └─ 支付超时 ─> 已超时(3)
```

## 13. 典型前端接入流程

### 13.1 用户秒杀流程

1. 调用 `GET /pc/seckill/user/activity/list` 获取活动列表
2. 调用 `GET /pc/seckill/user/activity/detail/{id}` 拉活动详情
3. 用户点击抢购，调用 `POST /pc/seckill/user/do/{activityId}/{itemId}`
4. 先拿到 `seckillNo`，状态一般是 `-1`
5. 前端轮询 `GET /pc/seckill/user/order/result/{activityId}/{itemId}`
6. 等返回 `orderNo` 与 `status=0`
7. 拿 `qrcode` / `payUrl` 发起支付
8. 支付页可继续轮询 `GET /pc/seckill/user/order/status/{seckillNo}`
9. 支付成功后状态变为 `1`

### 13.2 管理端配置流程

1. 商品预览：`GET /pc/seckill/goods/preview`
2. 商品入池：`POST /pc/seckill/goods/add`
3. 商品上架：`POST /pc/seckill/goods/status`
4. 创建活动草稿：`POST /pc/seckill/activity/add`
5. 发布活动：`POST /pc/seckill/activity/status`
6. 如需搜索展示同步到 ES：
   - 首次/索引异常：`POST /pc/seckill/es/recreateIndex`
   - 全量同步：`POST /pc/seckill/es/syncAll`

## 14. 代码层注意事项

### 14.1 商品类型定义存在“声明大于实现”的情况

- VO 与表结构里有 `3-商品`
- 但商品预览与入池逻辑当前只支持：
  - `1-课程`
  - `2-书籍`

### 14.2 `needPay` 字段当前未真正赋值

`SeckillResultVO` 里存在 `needPay`，但当前秒杀服务 `doSeckill()`、`toResultVO()` 中未显式设置该字段，因此前端不能依赖这个字段判断是否需要支付。

### 14.3 用户端动态接口返回的不是订单 VO

`GET /pc/seckill/user/recent/orders` 返回的是 `SeckillAnnouncementVO`，不是 `SeckillRecentOrderVO`。

### 14.4 `tagNameList` 只在 ES 路径下真正参与筛选

当 ES 关闭或异常降级到 MySQL 时，标签筛选条件当前会被忽略。

### 14.5 运维接口当前是匿名开放

以下接口当前带 `@Anonymous`：

- `/pc/seckill/user/esSync/all`
- `/pc/seckill/es/recreateIndex`
- `/pc/seckill/es/syncAll`

如果部署到生产环境，建议增加额外访问保护。
