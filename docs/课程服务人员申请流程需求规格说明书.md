# 课程服务人员申请流程需求规格说明书

## 一、业务流程概述

### 1.1 流程概述
用户通过考试与审核可申请成为课程服务人员，整个流程包括：**申请发起 → 准备考试 → 在线考试 → 成绩判定 → 管理员审批 → 结果通知**六个核心环节。

### 1.2 参与角色
- **申请人**：希望成为课程服务人员的普通用户
- **管理员**：负责审核服务人员申请的管理人员
- **系统**：自动处理考试、评分、状态流转等自动化操作

---

## 二、详细流程步骤

### 2.1 阶段一：申请发起与准备

#### 步骤 1：生成申请记录
**触发条件**：用户在课程详情页点击"申请成为服务人员"

**业务逻辑**：
```
1. 前端调用申请接口：POST /system/course/staff/apply
   - 参数：courseId（课程 ID）、staffType（服务类型：assistant/tutor/instructor）
   
2. 后端执行逻辑：
   - 检查用户是否已申请（查询 osh_course_staff 表）
     * 若 audit_status='approved' → 抛出异常"您已经是该课程的服务人员"
     * 若 audit_status='pending' → 抛出异常"您的申请正在审核中"
   - 保存申请记录到 osh_course_staff 表
     * 字段：user_id, course_id, staff_type, exam_score=null, 
            audit_status='pending', apply_time=NOW()
   - 返回申请 ID（applyId）
   
3. 前端跳转至准备考试界面
```

**数据流转**：
```
用户请求 → Controller → Service → Mapper → Database
  ↓
返回 applyId → 前端路由跳转
```

**状态变更**：
- `osh_course_staff.audit_status` = 'pending'（待审核）

---

### 2.2 阶段二：在线考试（异步调用）

#### 步骤 2.1：进入考试界面
**触发条件**：用户点击"开始考试"按钮

**业务逻辑**：
```
1. 前端调用考试详情接口：GET /pc/testpaper/read?id={examId}
   
2. 后端执行逻辑（OshExamServiceImpl.selectExamById）：
   - 查询试卷基本信息（标题、总分、及格分、时长等）
   - 查询题目列表（5 道题目，包含选项、正确答案）
   - 处理选项格式：JSON 字符串转为数组
   - 初始化答题数据（user_value 字段）
   - 返回 ExamDetailVo 对象
   
3. 前端渲染试卷界面，开始倒计时
```

**技术实现**：
- **同步调用**：获取试卷信息
- **异步标记**：考试过程独立于申请流程，不阻塞主线程

**数据示例**：
```json
{
  "id": 246,
  "title": "Java 后端核心技术能力评估（高级）",
  "total_score": 120,
  "pass_score": 72,
  "expire": 90,
  "testpaper_questions": [
    {
      "id": 1,
      "type": "radio",
      "options": ["A", "B", "C", "D"],
      "correct_answer": "A",
      "score": 10,
      "user_value": -1  // 默认值
    }
  ]
}
```

#### 步骤 2.2：提交答卷（异步处理）
**触发条件**：用户完成答题并点击"提交答卷"

**业务逻辑**：
```
1. 前端调用交卷接口：POST /pc/user_test/save
   - 参数：
     {
       "user_test_id": 7,  // 考试记录 ID
       "value": [          // 答案数组
         ["A"],            // 第 1 题答案
         ["正确"],          // 第 2 题答案
         ["多选 A","多选 B"] // 第 3 题答案（多选）
       ]
     }
   
2. 后端执行逻辑（OshUserExamRecordServiceImpl.saveUserExam）：
   【事务处理】
   a. 查询题目列表（按 ID 排序，确保与答案顺序一致）
   b. 遍历题目，自动批阅：
      - 客观题（单选/多选/判断/填空）：对比答案
        * userAnsStr == correctAnsStr → 累加分数
      - 主观题（简答）：跳过自动批阅，标记 needManualRead=1
   c. 计算最终得分 finalScore
   d. 更新考试记录表：
      - score = finalScore
      - answer_status = 1（已交卷）
      - read_status = isNeedManualRead ? 0 : 1
      - answer_json = JSON.toJSONString(userAnswers)
   
3. 异步回调：考试完成后触发后续流程
   - 调用课程模块接口：更新申请记录的考试成绩
   - 根据 pass_score 判断是否合格
```

**异步回调机制**：
```java
// 伪代码示例
@Async
public void handleExamCompleted(Long applyId, Integer examScore, Boolean isPassed) {
    // 1. 更新申请记录的成绩
    staffMapper.updateExamScore(applyId, examScore);
    
    // 2. 如果考试通过，自动进入待审核状态
    if (isPassed) {
        staffMapper.updateAuditStatus(applyId, "pending_review");
    } else {
        // 考试未通过，标记为拒绝
        staffMapper.updateAuditStatus(applyId, "rejected_exam");
        // 发送通知给用户
        notificationService.sendExamFailed(userId, courseId, examScore);
    }
}
```

**数据流转**：
```
用户提交答案 → Controller → Service（批阅） → Database
                    ↓
            异步回调 → CourseService → 更新申请记录
                    ↓
            判断成绩 → 发送通知
```

**状态变更**：
- `osh_user_exam_record.answer_status` = 1（已交卷）
- `osh_user_exam_record.read_status` = 1（已批阅）或 0（待人工批阅）
- `osh_course_staff.exam_score` = actualScore（实际得分）
- `osh_course_staff.audit_status` = 'qualified'（考试合格）或 'rejected_exam'（考试不合格）

### 2.3 阶段三：成绩判定与状态流转

#### 步骤 3：考试成绩关联与合格性判断
**触发条件**：考试批阅完成（自动或手动）

**业务逻辑**：
```
1. 系统自动获取考试结果：
   - 从 osh_user_exam_record 表查询 score 字段
   
2. 查询课程成绩标准：
   - 从 osh_examination 表查询 pass_score 字段
   
3. 判断合格性：
   IF score >= pass_score THEN
       isQualified = true
       newStatus = 'qualified'  // 考试合格，待审核
   ELSE
       isQualified = false
       newStatus = 'rejected_exam'  // 考试不合格，直接拒绝
   END IF
   
4. 更新申请记录：
   UPDATE osh_course_staff 
   SET exam_score = score, 
       audit_status = newStatus,
       update_time = NOW()
   WHERE id = applyId
   
5. 根据结果触发不同流程：
   - 合格 → 进入管理员审批队列
   - 不合格 → 发送失败通知，流程结束
```

**异常情况处理**：
| 异常情况 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 考试超时未完成 | 系统自动收卷，按已答题目计分 | "考试时间已到，系统已自动交卷" |
| 答案数量不匹配 | 拒绝提交，抛出 ServiceException | "提交的答案数量与题目总数不匹配" |
| 网络中断导致提交失败 | 本地暂存答案，恢复后重新提交 | "提交失败，答案已保存，请重试" |
| 考试过程中服务器宕机 | Redis 缓存答案，恢复后写入数据库 | "系统异常，答案已保护，请联系管理员" |

---

### 2.4 阶段四：管理员审批

#### 步骤 4.1：审批界面展示
**触发条件**：管理员登录后台管理系统

**业务逻辑**：
```
1. 管理员访问审批界面：GET /system/course/staff/applications
   - 参数：page, limit, auditStatus（筛选条件）
   
2. 后端查询申请列表：
   SELECT 
       s.id, s.user_id, s.course_id, s.staff_type, 
       s.exam_score, s.apply_time, s.audit_status,
       u.nick_name, u.avatar, u.dept_id,
       c.title AS course_title
   FROM osh_course_staff s
   LEFT JOIN sys_user u ON s.user_id = u.user_id
   LEFT JOIN osh_course c ON s.course_id = c.id
   WHERE s.audit_status = 'qualified'  -- 只显示考试合格的申请
   ORDER BY s.apply_time DESC
   
3. 返回申请列表数据（包含分页信息）
```

**界面元素**：
- 申请人信息：头像、昵称、部门
- 申请详情：课程名称、服务类型、考试成绩、申请时间
- 审批操作：通过/拒绝按钮、备注输入框
- 筛选功能：按课程、服务类型、申请时间筛选

---

#### 步骤 4.2：审批操作
**触发条件**：管理员点击"通过"或"拒绝"按钮

**业务逻辑**：
```
1. 前端调用审批接口：POST /system/course/staff/{applyId}/audit
   - 参数：
     {
       "auditStatus": "approved" 或 "rejected",
       "auditRemark": "同意录用为该课程助教" 或 "抱歉，您的申请未通过"
     }
   
2. 后端执行逻辑（CourseManageServiceImpl.auditStaff）：
   【事务处理】
   a. 验证管理员权限（@PreAuthorize("@ss.hasPermi('system:course:staff:audit')")）
   b. 更新审核状态：
      UPDATE osh_course_staff
      SET audit_status = #{auditStatus},
          audit_user_id = #{auditUserId},
          audit_time = NOW(),
          audit_remark = #{auditRemark},
          is_active = CASE WHEN #{auditStatus}='approved' THEN 1 ELSE 0 END,
          update_time = NOW()
      WHERE id = #{applyId}
   
3. 审批通过后执行后续操作：
   - 如果是"approved"：
     * 正式成为课程服务人员
     * 授予相应权限（system:course:question:answer 等）
     * 发送通过通知
   - 如果是"rejected"：
     * 发送拒绝通知
     * 允许重新申请（可选配置）
```

**状态变更**：
- `osh_course_staff.audit_status` = 'approved'（通过）或 'rejected'（拒绝）
- `osh_course_staff.audit_user_id` = 管理员 ID
- `osh_course_staff.audit_time` = 当前时间
- `osh_course_staff.audit_remark` = 审批备注
- `osh_course_staff.is_active` = 1（在职）或 0（离职）

---

### 2.5 阶段五：结果通知

#### 步骤 5：审批结果通知
**触发条件**：审批完成

**通知机制**：
```
1. 审批通过通知：
   【站内消息】
   - 标题："课程服务人员申请通过通知"
   - 内容：
     尊敬的{nick_name}：
     恭喜！您申请成为"{course_title}"课程{staffType}的请求已通过审核。
     您现在可以行使以下职责：
     - 回答课程问题
     - 管理课程资料
     - 编辑课程内容
     
     请立即前往课程管理页面查看详情。
     
   【邮件通知】（可选）
   - 发送至用户注册邮箱
   - 模板：email_templates/staff_approved.html
   
   【短信通知】（可选）
   - 内容：【XX 教育】恭喜您成为 XX 课程的服务人员，请登录系统查看。
   
2. 审批拒绝通知：
   【站内消息】
   - 标题："课程服务人员申请结果通知"
   - 内容：
     尊敬的{nick_name}：
     很遗憾，您申请成为"{course_title}"课程{staffType}的请求未通过审核。
     
     审核意见：{auditRemark}
     
     如有异议，请联系课程管理员或重新提交申请。
     
   【微信推送】（如果集成微信公众号）
   - 模板消息推送
```

**技术实现**：
```java
@Service
public class NotificationService {
    
    @Autowired
    private ISysNoticeService noticeService;
    
    @Autowired
    private IMailService mailService;
    
    /**
     * 发送审批通过通知
     */
    @Async
    public void sendApprovedNotification(Long userId, Long courseId, String staffType) {
        // 1. 查询用户信息
        SysUser oshUser = userService.selectUserById(userId);
        
        // 2. 查询课程信息
        OshCourse course = courseMapper.selectCourseById(courseId);
        
        // 3. 创建站内消息
        SysNotice notice = new SysNotice();
        notice.setNoticeTitle("课程服务人员申请通过");
        notice.setNoticeContent(buildApprovedContent(oshUser, course, staffType));
        notice.setNoticeType("1"); // 1=通知
        noticeService.insertNotice(notice);
        
        // 4. 插入用户通知关联表
        SysUserNotice userNotice = new SysUserNotice();
        userNotice.setUserId(userId);
        userNotice.setNoticeId(notice.getNoticeId());
        userNotice.setStatus("0"); // 0=未读
        userNoticeMapper.insertUserNotice(userNotice);
        
        // 5. 发送邮件（异步）
        if (StringUtils.isNotEmpty(oshUser.getEmail())) {
            mailService.sendApprovedEmail(oshUser.getEmail(), course.getTitle(), staffType);
        }
    }
    
    /**
     * 发送审批拒绝通知
     */
    @Async
    public void sendRejectedNotification(Long userId, Long courseId, String auditRemark) {
        // 类似 approved 逻辑...
    }
}
```

## 六、异常情况处理规范

### 6.1 考试环节异常

| 异常场景 | 触发条件 | 处理逻辑 | 用户提示 |
|---------|---------|---------|---------|
| 考试超时 | 超过 expire 分钟未交卷 | 系统自动调用 saveUserTest，按已答题目计分 | "考试时间已到，系统已自动交卷" |
| 答案缺失 | value 数组长度 < 题目数量 | 抛出 ServiceException，拒绝提交 | "提交的答案不完整，请检查后重试" |
| 重复交卷 | answer_status 已为 1 | 检查是否允许重新考试，否则拒绝 | "您已经交过卷了，无法重复提交" |
| 网络中断 | 提交过程中连接断开 | 前端本地存储（localStorage）答案，恢复后重交 | "提交失败，答案已保存，网络恢复后将自动提交" |
| 服务器宕机 | 批阅过程中服务崩溃 | Redis 缓存答案，重启后继续批阅 | "系统维护中，请稍后再试" |

### 6.2 审批环节异常

| 异常场景 | 触发条件 | 处理逻辑 | 用户提示 |
|---------|---------|---------|---------|
| 重复审批 | 同一 applyId 被多次审批 | 检查 audit_status 是否已为 terminal state，是则拒绝 | "该申请已审核完毕，无法重复操作" |
| 越权审批 | 非管理员尝试审批 | Spring Security 拦截，返回 403 | "您没有权限执行此操作" |
| 审批备注为空（拒绝时） | auditStatus='rejected' 但 auditRemark 为空 | 后端校验，拒绝为空必填项 | "拒绝申请时必须填写审核意见" |
| 审批后权限未授予 | approved 但用户无 perms | 检查权限同步机制，手动触发授权 | "系统延迟，请稍后刷新权限" |

### 6.3 通知环节异常

| 异常场景 | 触发条件 | 处理逻辑 | 用户提示 |
|---------|---------|---------|---------|
| 邮件发送失败 | SMTP 服务器不可用 | 记录错误日志，重试 3 次，仍失败则仅发送站内消息 | 无感知（降级处理） |
| 用户未绑定手机/邮箱 | 联系方式为空 | 跳过对应渠道通知，仅发送站内消息 | 无感知 |
| 消息队列堵塞 | 通知量过大 | 异步队列削峰填谷，延迟发送 | 稍有延迟 |

## 七、数据安全与隐私保护

### 7.1 敏感信息加密
- **考试答案**：answer_json 字段 AES 加密存储
- **用户 ID**：传输过程使用 JWT Token 验证
- **成绩信息**：仅本人和管理员可见

### 7.2 防作弊机制
- **题目乱序**：每次考试题目顺序随机打乱
- **选项乱序**：选择题选项顺序随机
- **切屏监控**：记录切屏次数，超过阈值强制交卷（可选）
- **IP 限制**：同一 IP 只能考试一次（防止替考）

## 八、性能优化建议

### 8.1 数据库优化
- **索引设计**：
  ```sql
  CREATE INDEX idx_audit_status ON osh_course_staff(audit_status);
  CREATE INDEX idx_user_course ON osh_course_staff(user_id, course_id);
  CREATE INDEX idx_exam_score ON osh_user_exam_record(score);
  ```
  
- **分区表**（数据量>100 万时考虑）：
  ```sql
  ALTER TABLE osh_course_staff 
  PARTITION BY RANGE (YEAR(apply_time)) (
      PARTITION p2025 VALUES LESS THAN (2026),
      PARTITION p2026 VALUES LESS THAN (2027),
      ...
  );
  ```

### 8.2 缓存策略
- **试卷信息缓存**：Redis 缓存 5 分钟，减少数据库查询
  ```java
  @Cacheable(value = "exam_paper", key = "#examId", expire = 300)
  public ExamDetailVo selectExamById(Long examId)
  ```
  
- **申请状态缓存**：实时性要求高，不建议缓存

### 8.3 异步处理
- **考试批阅**：@Async 异步批阅，快速响应用户
- **通知发送**：消息队列（RabbitMQ/RocketMQ）削峰
- **统计计算**：定时任务批量计算通过率
