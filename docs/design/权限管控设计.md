# 权限管控设计

本文说明本项目前后端实际使用的两套权限管控机制：后端接口权限拦截和前端页面/按钮权限控制。两者使用同一份角色、菜单、权限字符数据，但职责不同：

- 后端权限是最终安全边界，负责拦截接口调用。
- 前端权限是用户体验层，负责控制菜单、路由、按钮是否显示。

前端隐藏按钮不能替代后端校验。任何新增、修改、删除、审核等敏感接口，都必须在后端加权限注解。

## 一、权限数据模型

系统权限主体沿用 RuoYi 的 RBAC 模型：

- 用户：`sys_user`
- 角色：`sys_role`
- 用户角色关系：`sys_user_role`
- 菜单/按钮权限：`sys_menu`
- 角色菜单关系：`sys_role_menu`

`sys_menu` 中最关键的字段是：

- `menu_type`：菜单类型。`M` 表示目录，`C` 表示菜单，`F` 表示按钮权限。
- `perms`：权限字符，例如 `course:create`、`system:course:edit`、`tool:vote:good`。
- `path`、`component`：前端路由地址和组件路径。
- `visible`、`status`：控制菜单是否显示、是否启用。

角色绑定菜单后，用户会间接拥有这些菜单对应的权限字符。后端和前端都以这些权限字符作为判断依据。

管理员有特殊逻辑：

- 后端菜单权限集合会包含 `*:*:*`，表示拥有所有按钮权限。
- 前端角色集合会包含 `admin`，表示拥有所有角色权限。

## 二、后端权限管控设计

### 1. 登录态和权限加载

用户登录后，后端生成 token。之后前端请求 `GET /getInfo` 获取当前用户信息、角色集合和权限集合。

核心流程如下：

1. 用户调用 `/login` 登录，后端返回 token。
2. 前端保存 token，后续请求在请求头中携带 token。
3. `JwtAuthenticationTokenFilter` 从 token 中解析 `LoginUser`。
4. 过滤器把 `LoginUser` 放入 Spring Security 的 `SecurityContextHolder`。
5. 业务接口执行前，Spring Security 根据 `@PreAuthorize` 做权限判断。

`LoginUser` 中保存了当前登录用户的权限集合：

```java
private Set<String> permissions;
```

`SysLoginController#getInfo` 会调用：

```java
Set<String> roles = permissionService.getRolePermission(user);
Set<String> permissions = permissionService.getMenuPermission(user);
```

其中：

- `getRolePermission` 从用户角色中取角色标识，例如 `admin`。
- `getMenuPermission` 从角色绑定的菜单中取 `sys_menu.perms`。
- 如果用户是管理员，则直接返回 `*:*:*`。

### 2. 第一种后端写法：`@ss.hasPermi`

这是 RuoYi 原生的权限写法：

```java
@PreAuthorize("@ss.hasPermi('system:course:edit')")
```

`@ss` 对应后端 Bean：

```java
@Service("ss")
public class PermissionService
```

判断逻辑是：

1. 从 `SecurityUtils.getLoginUser()` 取当前登录用户。
2. 读取 `loginUser.getPermissions()`。
3. 判断权限集合中是否包含目标权限字符。
4. 如果权限集合中包含 `*:*:*`，则直接放行。

适用场景：

- 后台管理类接口。
- 与 `sys_menu.perms` 强绑定的接口。
- 需要和前端 `v-hasPermi` 使用同一个权限字符的接口。

课程模块中已有示例：

```java
@PreAuthorize("@ss.hasPermi('system:course:section:add')")
@PostMapping("/{courseId}/section")
public R<Long> addSection(...)
```

表示：只有拥有 `system:course:section:add` 权限的用户，才可以调用添加章节/课时接口。

### 3. 第二种后端写法：`hasAuthority`

项目里也使用了 Spring Security 原生写法：

```java
@PreAuthorize("hasAuthority('course:create')")
```

它直接判断当前 Authentication 的 authorities 中是否存在指定权限。当前项目的 `LoginUser#getAuthorities()` 会把 `permissions` 转换成 Spring Security 可识别的权限集合，所以它和 `@ss.hasPermi` 的底层权限来源一致，都是当前登录用户拥有的权限字符。

课程模块中已有示例：

```java
@ApiOperation("新增/修改课程")
@PostMapping("/save")
@PreAuthorize("hasAuthority('course:create')")
public R<Long> save(@RequestBody CourseCreateRequest request) {
    ...
}
```

表示：只有拥有 `course:create` 权限的用户，才可以新增课程。

再比如：

```java
@ApiOperation("修改课程")
@PostMapping("/update")
@PreAuthorize("hasAuthority('course:update')")
public R<Long> update(@Validated @RequestBody CourseUpdateRequest request) {
    ...
}
```

表示：只有拥有 `course:update` 权限的用户，才可以修改课程。

### 4. 两种后端写法的区别和建议

两种写法最终都依赖用户权限集合，但表达方式不同：

- `@ss.hasPermi('xxx')`：项目封装写法，可读性更贴近菜单权限，支持 `hasAnyPermi`、`hasRole` 等自定义能力。
- `hasAuthority('xxx')`：Spring Security 原生写法，直接判断 authority。

建议新代码优先使用统一风格。后台管理模块建议使用：

```java
@PreAuthorize("@ss.hasPermi('模块:资源:动作')")
```

如果某个业务模块已经大量使用 `hasAuthority('course:*')`，短期可以保持一致，但必须确保 `sys_menu.perms` 中配置了完全一致的权限字符。

注意：权限字符必须精确匹配。`course:create` 和 `system:course:add` 是两个完全不同的权限。

## 三、前端权限管控设计

前端权限主要分两层：

- 路由权限：控制用户能看到哪些菜单、进入哪些页面。
- 按钮权限：控制页面中的新增、修改、删除、审核等按钮是否显示。

### 1. 前端权限加载流程

登录后，前端流程如下：

1. `src/permission.js` 路由守卫判断本地是否有 token。
2. 如果有 token 但 Vuex 中还没有角色信息，则调用 `store.dispatch('GetInfo')`。
3. `GetInfo` 请求后端 `/getInfo`，拿到 `roles` 和 `permissions`。
4. Vuex 保存：

```js
roles: []
permissions: []
```

5. 前端再调用 `store.dispatch('GenerateRoutes')` 请求 `/getRouters`。
6. 后端根据用户角色返回可访问菜单树。
7. 前端把菜单树转换成 Vue Router 路由，并挂载到侧边栏。

因此，菜单级权限主要由后端 `/getRouters` 决定；按钮级权限主要由前端指令读取 Vuex 中的 `permissions` 决定。

### 2. 路由权限

后端接口：

```java
@GetMapping("getRouters")
public AjaxResult getRouters() {
    Long userId = SecurityUtils.getUserId();
    List<SysMenu> menus = menuService.selectMenuTreeByUserId(userId);
    return AjaxResult.success(menuService.buildMenus(menus));
}
```

后端会根据用户角色查询 `sys_menu`，再构造成前端可识别的路由结构。

前端 `src/store/modules/permission.js` 中的 `GenerateRoutes` 会：

- 请求 `/getRouters`。
- 把后端返回的 `component` 字符串转换成真实 Vue 组件。
- 设置侧边栏路由 `sidebarRouters`。
- 动态添加需要权限控制的本地隐藏路由。

如果某个页面是后端菜单管理中配置的菜单，例如课程管理页面，则用户只有被分配该菜单后才能在侧边栏看到入口。

### 3. 按钮权限：`v-hasPermi`

项目中最常用的按钮权限指令是：

```vue
<el-button v-hasPermi="['system:user:add']">新增</el-button>
```

实现位置：

```text
backstage-ui/src/directive/permission/hasPermi.js
```

判断逻辑是：

1. 从 Vuex 读取 `store.getters.permissions`。
2. 判断用户权限中是否包含按钮要求的权限字符。
3. 如果用户没有权限，则直接从 DOM 中移除该按钮。
4. 如果用户拥有 `*:*:*`，则所有按钮都显示。

示例：

```vue
<el-button
  type="primary"
  icon="el-icon-plus"
  v-hasPermi="['course:create']"
  @click="handleCreate"
>
  新增课程
</el-button>
```

表示：只有拥有 `course:create` 权限的用户才能看到新增课程按钮。

### 4. 角色权限：`v-hasRole`

角色权限指令用于按角色控制显示：

```vue
<el-button v-hasRole="['admin']">管理员操作</el-button>
```

实现位置：

```text
backstage-ui/src/directive/permission/hasRole.js
```

它从 Vuex 读取 `roles`，如果用户拥有目标角色，或者拥有 `admin` 超级角色，则显示元素。

角色权限适合少量强角色判断，例如“仅管理员可见”。业务按钮通常更建议用 `v-hasPermi`，因为权限字符可以精细到具体动作。

### 5. JS 中手动判断权限

除指令外，项目还提供了工具函数：

```js
import { checkPermi, checkRole } from '@/utils/permission'
```

适用场景：

- 需要在 `computed` 中控制某个区域显示。
- 需要在 JS 方法里提前判断是否允许执行。
- 需要控制表格列、下拉项、复杂组件内部状态。

示例：

```js
computed: {
  canCreateCourse() {
    return checkPermi(['course:create'])
  }
}
```

但即使前端手动判断了权限，后端接口仍然必须加 `@PreAuthorize`。

## 四、课程模块权限示例

课程模块可以用来理解完整链路：菜单配置、前端显示、后端拦截必须保持同一套权限字符。

### 1. 权限字符设计

课程模块可以按动作拆分权限：

| 功能 | 权限字符 | 用途 |
| --- | --- | --- |
| 课程列表/搜索 | `course:list` | 查询课程列表、ES 搜索 |
| 新增课程 | `course:create` | 创建课程 |
| 修改课程 | `course:update` | 修改课程、审核课程 |
| 删除课程 | `course:delete` | 删除课程、删除章节 |
| 新增/修改章节 | `course:chapter:save` | 保存章节 |
| 新增视频小节 | `course:section:video` | 添加视频小节 |
| 新增文本小节 | `course:section:text` | 添加文本小节 |
| 提交课程提问 | `course:question:submit` | 学员提交问题 |
| 回答课程问题 | `course:question:answer` | 服务人员或管理员回答问题 |

如果使用后台管理命名，也可以设计成：

| 功能 | 权限字符 |
| --- | --- |
| 课程管理菜单 | `system:course:list` |
| 修改课程 | `system:course:edit` |
| 删除课程 | `system:course:remove` |
| 添加章节/课时 | `system:course:section:add` |
| 回答问题 | `system:course:question:answer` |
| 审核服务人员 | `system:course:staff:audit` |

关键要求：前端按钮、后端注解、`sys_menu.perms` 必须使用同一个字符串。

### 2. 后端接口控制

新增课程接口：

```java
@PostMapping("/save")
@PreAuthorize("hasAuthority('course:create')")
public R<Long> save(@RequestBody CourseCreateRequest request) {
    ...
}
```

修改课程接口：

```java
@PostMapping("/update")
@PreAuthorize("hasAuthority('course:update')")
public R<Long> update(@Validated @RequestBody CourseUpdateRequest request) {
    ...
}
```

删除课程接口：

```java
@PostMapping("/delete")
@PreAuthorize("hasAuthority('course:delete')")
public R<String> deleteCourses(@RequestBody CourseDeleteRequest request) {
    ...
}
```

后台管理风格接口：

```java
@PreAuthorize("@ss.hasPermi('system:course:section:add')")
@PostMapping("/{courseId}/section")
public R<Long> addSection(...) {
    ...
}
```

这些注解会在方法执行前拦截请求。用户没有权限时，业务代码不会执行。

### 3. 前端按钮控制

课程列表页中，新增课程按钮应该写成：

```vue
<el-button
  type="primary"
  icon="el-icon-plus"
  v-hasPermi="['course:create']"
  @click="handleCreate"
>
  新增课程
</el-button>
```

删除按钮应该写成：

```vue
<el-dropdown-item
  command="delete"
  icon="el-icon-delete"
  v-hasPermi="['course:delete']"
>
  删除课程
</el-dropdown-item>
```

如果课程页面使用的是后台管理权限字符，则前端也要改成同一套：

```vue
<el-button v-hasPermi="['system:course:edit']">修改课程</el-button>
```

不要出现前端用 `course:update`，后端用 `system:course:edit`，菜单表里又配置第三个名字的情况。这样会导致前端显示和后端拦截不一致。

### 4. 当前课程演示页的特殊情况

当前 `backstage-ui/src/views/course/**` 下的课程页面属于演示路由：

```js
const whiteList = ['/login', '/register', '/demo/**']
```

并且课程列表页目前使用的是：

```js
import { hasPermission } from '@/api/course/courseListMock'
```

这是一套 mock 权限，用来演示不同角色下按钮是否显示，不是正式后台权限体系。

课程模块接入正式权限时，需要做三件事：

1. 去掉演示路由白名单中的课程入口，改为后端菜单返回的正式路由。
2. 页面按钮从 `hasPermission('course:create')` 改为 `v-hasPermi="['course:create']"` 或 `checkPermi(['course:create'])`。
3. 后端接口保留或补齐 `@PreAuthorize`，确保接口层有最终拦截。

## 五、新增一个权限点的落地步骤

以“课程审核”为例，完整落地步骤如下。

### 1. 设计权限字符

先确定权限字符，例如：

```text
course:audit
```

或者如果走后台管理风格：

```text
system:course:audit
```

同一个功能只选一个，避免混用。

### 2. 在菜单管理中新增按钮权限

在系统管理、菜单管理中，找到课程管理菜单，新增按钮类型权限：

- 菜单类型：按钮
- 按钮名称：课程审核
- 权限字符：`course:audit`
- 状态：正常

数据库层面本质是向 `sys_menu` 增加一条 `menu_type = 'F'` 的记录，并通过 `sys_role_menu` 分配给角色。

### 3. 给角色分配权限

进入角色管理，给需要审核课程的角色勾选“课程审核”按钮权限。

分配后，该角色下的用户重新登录或重新拉取 `/getInfo` 后，会拿到新的权限字符。

### 4. 后端接口加注解

```java
@PostMapping("/audit")
@PreAuthorize("hasAuthority('course:audit')")
public R<Long> audit(@Validated @RequestBody CourseAuditRequest request) {
    ...
}
```

如果采用 `@ss.hasPermi` 风格：

```java
@PreAuthorize("@ss.hasPermi('course:audit')")
```

### 5. 前端按钮加指令

```vue
<el-button
  type="warning"
  icon="el-icon-check"
  v-hasPermi="['course:audit']"
  @click="handleAudit(row)"
>
  审核
</el-button>
```

这样没有权限的用户看不到审核按钮；即使用户绕过前端直接请求接口，后端也会拒绝。

## 六、常见注意事项

### 1. 前端权限不是安全边界

`v-hasPermi` 只是从 DOM 中移除按钮，用户仍然可能通过浏览器控制台、抓包工具或直接调用接口绕过前端。因此所有敏感接口都必须加后端权限注解。

### 2. 权限字符必须统一

同一个操作的权限字符要在三个地方完全一致：

- `sys_menu.perms`
- 后端 `@PreAuthorize`
- 前端 `v-hasPermi` 或 `checkPermi`

### 3. `@Anonymous` 和 `@PreAuthorize` 不要混用

如果一个接口标了 `@Anonymous`，表示允许匿名访问。敏感接口不应该同时标匿名和权限注解。课程模块中存在个别接口同时出现 `@Anonymous` 和 `@PreAuthorize` 的情况，后续整理时建议明确接口定位：

- 公开查询接口：保留 `@Anonymous`，不要加权限注解。
- 管理操作接口：去掉 `@Anonymous`，保留 `@PreAuthorize`。

### 4. 角色权限不要滥用

`v-hasRole` 和 `@ss.hasRole` 适合判断“管理员”“运营人员”这类大角色。具体按钮动作建议使用权限字符，例如 `course:delete`，这样后续可以给不同角色灵活分配。

### 5. 权限变更后需要刷新登录态

用户登录后权限会进入 token/前端 Vuex。角色权限调整后，用户需要重新登录，或者后端刷新 token 中的 `LoginUser.permissions`，前端重新调用 `/getInfo`，才能看到最新权限。

## 七、推荐规范

后续新增模块建议采用统一格式：

```text
模块:资源:动作
```

示例：

```text
course:list
course:create
course:update
course:delete
course:question:answer
tool:vote:good
tool:vote:bad
```

后台系统管理类模块也可以沿用 RuoYi 风格：

```text
system:user:add
system:role:edit
system:course:remove
```

无论选择哪种风格，模块内部要保持一致。新增接口时先定权限字符，再同步配置菜单、后端注解和前端按钮。
