# 新增/修改/删除课程 + 本地消息表保证 Kafka 消息可靠发送设计说明

## 1. 设计背景

课程新增、修改、删除后，需要同步通知下游系统更新课程搜索索引。

当前链路是：

```text
课程业务表变更 -> 写入本地消息表 -> Kafka -> Flink -> ES
```

这里最核心的问题是：课程主业务数据写入数据库成功后，Kafka 消息也必须尽量可靠地发送出去。

如果直接在业务方法里调用 Kafka，会遇到这些问题：

1. 课程数据保存成功了，但 Kafka 发送失败，导致 ES 没有同步。
2. Kafka 发送成功了，但本地事务回滚，导致下游收到一条实际不存在的课程数据。
3. 服务在事务提交后、Kafka 发送前宕机，消息丢失。
4. Kafka 临时不可用时，业务接口不应该因为消息发送失败而整体不可用。

所以当前设计引入了本地消息表，也就是 Outbox 模式。

## 2. 总体设计思路

课程新增、修改、删除时，不再把 Kafka 发送作为主业务事务的一部分直接处理，而是先把“需要发送的消息”保存到本地数据库表。

本地消息表和课程业务表共用同一个数据库事务。

也就是说：

```text
课程表变更成功 + 本地消息表写入成功 -> 一起提交
课程表变更失败 -> 本地消息表也回滚
```

事务提交成功后，系统会立即尝试发送 Kafka 消息。

如果立即发送成功，则本地消息表状态变成已发送。

如果立即发送失败，则不会影响课程业务结果，消息会保留在本地消息表中，后续由定时任务继续扫描并重试发送。

整体效果：

```text
主业务事务保证消息不丢
事务提交后立即发送保证实时性
定时任务兜底保证最终送达
```

## 3. 核心流程

### 3.1 新增课程流程

```text
1. 校验课程新增参数
2. 插入课程主表
3. 绑定课程资料
4. 绑定课程标签
5. 查询最新课程数据
6. 构造课程索引新增消息
7. 写入本地消息表，状态为 PENDING
8. 本地事务提交
9. 事务提交后立即发送 Kafka
10. 发送成功则标记 SENT，发送失败等待定时任务重试
```

对应业务含义：

```text
新增课程成功后，下游需要新增 ES 中的课程索引文档。
```

消息类型：

```text
COURSE_INDEX_CREATE
```

Kafka topic：

```text
osh.course.index.create
```

示例代码逻辑：

```java
@Transactional(rollbackFor = Exception.class)
public Long createCourse(CourseCreateRequest request, OshUser operator) {
    OshCourse course = buildCourseForCreate(request, operator);
    oshCourseMapper.insertCourse(course);

    bindCourseMaterial(course.getId(), request.getMaterial(), operator);
    bindCourseTags(course.getId(), request.getTags(), operator);

    OshCourse latestCourse = ensureCourseExists(course.getId());

    outboxEventService.saveCourseIndexCreateEvent(
        course.getId(),
        buildCourseIndexUpsertMessage(latestCourse, request.getTags(), operator),
        operator
    );

    return course.getId();
}
```

### 3.2 修改课程流程

```text
1. 校验课程是否存在
2. 校验当前操作人是否有权限修改
3. 更新课程主表
4. 如有资料变更，重建课程资料
5. 如有标签变更，重建课程标签
6. 查询最新课程数据
7. 构造课程索引修改消息
8. 写入本地消息表，状态为 PENDING
9. 本地事务提交
10. 事务提交后立即发送 Kafka
11. 发送失败后由定时任务兜底重试
```

对应业务含义：

```text
修改课程成功后，下游需要更新 ES 中已有课程索引文档。
```

消息类型：

```text
COURSE_INDEX_UPDATE
```

Kafka topic：

```text
osh.course.index.update
```

示例代码逻辑：

```java
@Transactional(rollbackFor = Exception.class)
public Long updateCourse(CourseUpdateRequest request, OshUser operator) {
    OshCourse existingCourse = ensureCourseExists(request.getId());
    ensureCourseEditableByOperator(existingCourse, operator);

    OshCourse course = buildCourseForUpdate(request, operator);
    oshCourseMapper.updateCourse(course);

    if (request.getMaterial() != null) {
        rebuildCourseMaterial(course.getId(), request.getMaterial(), operator);
    }

    if (request.getTags() != null) {
        rebuildCourseTags(course.getId(), request.getTags(), operator);
    }

    OshCourse latestCourse = ensureCourseExists(course.getId());

    outboxEventService.saveCourseIndexUpdateEvent(
        course.getId(),
        buildCourseIndexUpsertMessage(latestCourse, request.getTags(), operator),
        operator
    );

    return course.getId();
}
```

### 3.3 删除课程流程

```text
1. 校验课程是否存在
2. 软删除课程主表
3. 软删除课程章节和小节
4. 删除课程标签关联
5. 删除课程资料
6. 构造课程索引删除消息
7. 写入本地消息表，状态为 PENDING
8. 本地事务提交
9. 事务提交后立即发送 Kafka
10. 发送失败后由定时任务兜底重试
```

对应业务含义：

```text
删除课程成功后，下游需要删除 ES 中对应课程索引文档。
```

消息类型：

```text
COURSE_INDEX_DELETE
```

Kafka topic：

```text
osh.course.index.delete
```

删除消息只需要课程 ID：

```java
new CourseIndexDeleteMessage(courseId)
```

这样可以减少删除链路和课程完整字段之间的耦合。

## 4. 本地消息表设计

本地消息表用于保存待发送 Kafka 的事件。

核心字段含义如下：

| 字段 | 说明 |
|---|---|
| `id` | 本地消息表主键 |
| `event_id` | 全局事件 ID，使用 UUID 生成 |
| `aggregate_type` | 聚合类型，课程场景固定为 `COURSE` |
| `aggregate_id` | 聚合 ID，课程场景就是 courseId |
| `event_type` | 事件类型，例如新增、修改、删除课程索引 |
| `topic` | Kafka topic |
| `message_key` | Kafka message key，当前为 `course:{courseId}` |
| `payload` | Kafka 消息内容，JSON 格式 |
| `status` | 消息状态 |
| `retry_count` | 当前重试次数 |
| `max_retry_count` | 最大重试次数，当前默认 10 |
| `next_retry_time` | 下次允许重试时间 |
| `last_error` | 最近一次失败原因 |
| `sent_time` | 成功发送时间 |
| `create_by/create_time` | 创建人和创建时间 |
| `update_by/update_time` | 更新人和更新时间 |
| `delete_flag` | 逻辑删除标记 |

## 5. 消息状态流转

当前消息状态定义：

| 状态值 | 状态名 | 含义 |
|---|---|---|
| `0` | `PENDING` | 待发送 |
| `1` | `SENDING` | 发送中 |
| `2` | `SENT` | 已发送 |
| `3` | `DEAD` | 死信，超过最大重试次数 |

正常流转：

```text
PENDING -> SENDING -> SENT
```

发送失败流转：

```text
PENDING -> SENDING -> PENDING
```

超过最大重试次数：

```text
PENDING -> SENDING -> DEAD
```

如果服务在 `SENDING` 状态时宕机，定时任务会把超时的 `SENDING` 消息恢复成 `PENDING`：

```text
SENDING -> PENDING
```

当前超时时间是 10 分钟。

## 6. 事务提交后立即发送

本地消息表写入成功后，系统不会马上在事务内部发送 Kafka，而是注册事务提交后的回调。

核心逻辑：

```java
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
    @Override
    public void afterCommit() {
        publishSafely(eventId);
    }
});
```

这样设计的原因是：

```text
只有本地事务真正提交成功后，Kafka 消息才允许发送。
```

如果事务回滚：

```text
课程数据回滚
本地消息表回滚
afterCommit 不执行
Kafka 不发送
```

这可以避免“下游收到消息，但本地数据库没有对应数据”的问题。

如果当前没有事务同步环境，则会直接尝试发送：

```java
publishSafely(eventId);
```

## 7. 立即发送失败后的兜底机制

事务提交后立即发送 Kafka 时，使用了安全发送逻辑：

```java
try {
    outboxEventPublishTask.publishEventById(eventId);
} catch (Exception ex) {
    log.warn("提交后立即投递outbox事件异常，等待定时任务兜底");
}
```

也就是说：

```text
立即发送失败不会影响主业务接口结果。
```

失败原因可能是：

- Kafka 临时不可用
- 网络异常
- Kafka broker 抖动
- 发送超时
- 服务实例正在关闭

只要本地消息表中还存在 `PENDING` 消息，定时任务就会继续扫描并发送。

## 8. 定时任务兜底发送

当前定时任务每 3 秒执行一次：

```java
@Scheduled(fixedDelay = 3000)
public void publishPendingEvents() {
    recoverTimeoutSendingEvents();
    List<OshOutboxEvent> events = outboxEventMapper.selectPendingEvents(100);
    for (OshOutboxEvent event : events) {
        publishEvent(event);
    }
}
```

每轮处理逻辑：

```text
1. 先恢复超时的 SENDING 消息
2. 查询最多 100 条 PENDING 消息
3. 按 id 升序发送
4. 发送前先把状态从 PENDING 改成 SENDING
5. 发送成功后标记 SENT
6. 发送失败则增加 retry_count，计算 next_retry_time
7. 超过最大重试次数则进入 DEAD
```

## 9. 并发安全设计

定时任务可能在多个服务节点同时运行。

为了避免同一条消息被多个节点重复发送，发送前会先抢占消息：

```sql
UPDATE osh_outbox_event
SET status = 1
WHERE id = #{id}
  AND status = 0
  AND next_retry_time <= NOW()
  AND delete_flag = 0
```

只有更新成功的节点才有资格发送。

效果：

```text
多个节点同时扫描到同一条 PENDING 消息
只有一个节点能把它改成 SENDING
其他节点更新失败，跳过发送
```

这保证了发送任务层面的并发安全。

## 10. 重试策略

当前默认最大重试次数：

```text
10 次
```

失败后的下次重试时间：

| retry_count | 下次重试时间 |
|---|---|
| 1 | 1 分钟后 |
| 2 | 5 分钟后 |
| 3 | 15 分钟后 |
| 4 及以上 | 1 小时后 |

这样设计的好处：

```text
短暂故障快速恢复
持续故障降低无效重试频率
避免 Kafka 长时间异常时把系统打满
```

当重试次数达到最大次数后，消息进入 `DEAD` 状态，需要人工排查。

## 11. 为什么不直接在业务事务里发 Kafka

不推荐：

```text
开启事务
修改课程表
发送 Kafka
提交事务
```

问题是 Kafka 和 MySQL 不在同一个本地事务里。

可能出现：

```text
Kafka 发送成功，MySQL 提交失败
```

下游会处理一条不存在或未成功变更的数据。

也不推荐：

```text
开启事务
修改课程表
提交事务
发送 Kafka
```

问题是：

```text
MySQL 提交成功，Kafka 发送前服务宕机
```

消息会丢失。

当前 Outbox 方案：

```text
开启事务
修改课程表
写本地消息表
提交事务
事务提交后立即发 Kafka
失败后定时任务兜底
```

解决的是：

```text
业务数据和消息记录的一致性
```

## 12. 当前方案能保证什么

### 12.1 保证课程数据成功后消息不会丢

因为课程变更和消息记录在同一个本地事务中。

只要课程事务提交成功，消息表里就一定有一条待发送记录。

### 12.2 保证事务回滚时不会发送消息

Kafka 发送注册在 `afterCommit` 中。

事务没提交，就不会触发立即发送。

### 12.3 保证发送失败后可以继续重试

立即发送失败后，消息仍然保留在本地消息表中。

定时任务会继续扫描并重试。

### 12.4 保证多节点下尽量避免重复发送

通过状态抢占：

```text
PENDING -> SENDING
```

同一条消息同一时间只会被一个节点处理。

### 12.5 保证发送实时性

不是只靠定时任务慢慢扫。

事务提交后会立即调用：

```java
publishEventById(eventId)
```

所以正常情况下，课程变更提交后 Kafka 会马上收到消息。

定时任务只是兜底。

## 13. 当前方案不能保证什么

### 13.1 不能保证 Kafka 绝对只收到一次

如果 Kafka 发送成功，但服务在标记 `SENT` 前宕机，这条消息后续可能会被重新发送。

所以整体语义更接近：

```text
至少发送一次
```

下游 Flink 或 ES 写入逻辑需要按课程 ID 做幂等处理。

例如：

- 新增/修改索引用 courseId 覆盖写
- 删除索引用 courseId 删除
- 消费端不要依赖消息只出现一次

### 13.2 不能替代业务校验

Outbox 只保证消息可靠发送，不负责判断课程是否可以新增、修改、删除。

课程权限、状态、参数合法性仍然要在业务层校验。

### 13.3 不能自动处理 DEAD 消息

超过最大重试次数后，消息会进入 `DEAD`。

这类消息说明长期发送失败，需要人工排查：

- Kafka topic 是否存在
- Kafka 集群是否正常
- payload 是否非法
- 下游是否兼容当前消息格式
- 是否需要手动重置状态重新发送

## 14. 设计效果举例

### 14.1 新增课程成功，Kafka 正常

```text
课程表插入成功
outbox 插入 COURSE_INDEX_CREATE
事务提交
afterCommit 立即发送 Kafka
发送成功
outbox 状态改为 SENT
Flink 消费消息
ES 新增课程文档
```

最终效果：

```text
数据库有课程
ES 有课程索引
消息状态为 SENT
```

### 14.2 新增课程失败，事务回滚

```text
课程表插入失败
事务回滚
outbox 也回滚
afterCommit 不执行
Kafka 不发送
```

最终效果：

```text
数据库没有课程
ES 不会新增错误索引
消息表没有无效消息
```

### 14.3 修改课程成功，但 Kafka 临时失败

```text
课程表更新成功
outbox 插入 COURSE_INDEX_UPDATE
事务提交
afterCommit 立即发送 Kafka
Kafka 发送失败
消息保留为 PENDING 或重试状态
定时任务后续扫描
Kafka 恢复后发送成功
状态改为 SENT
```

最终效果：

```text
接口不受 Kafka 临时故障影响
消息最终仍可发送出去
ES 最终完成更新
```

### 14.4 删除课程成功，服务提交后宕机

```text
课程软删除成功
章节、标签、资料处理完成
outbox 插入 COURSE_INDEX_DELETE
事务提交
服务在发送 Kafka 前宕机
```

服务恢复后：

```text
定时任务扫描 PENDING 消息
发送删除消息到 Kafka
Flink 删除 ES 文档
状态改为 SENT
```

最终效果：

```text
即使服务宕机，删除消息也不会丢
```

### 14.5 多节点同时扫描同一条消息

```text
server-1 扫描到 eventId=abc
server-2 也扫描到 eventId=abc
```

两个节点同时执行抢占：

```sql
UPDATE osh_outbox_event
SET status = 1
WHERE id = ?
  AND status = 0
```

结果：

```text
只有一个节点更新成功
另一个节点更新行数为 0，跳过
```

最终效果：

```text
避免同一条消息被多个节点同时发送
```

## 15. 推荐后续关注点

### 15.1 下游消费必须幂等

当前方案是至少一次投递，不是严格一次投递。

建议下游按照 `courseId` 幂等处理：

```text
CREATE / UPDATE：按 courseId upsert ES 文档
DELETE：按 courseId delete ES 文档
```

### 15.2 DEAD 消息需要监控

建议后续增加：

- DEAD 消息告警
- 后台管理页面
- 手动重试入口
- 按 eventId 查询发送历史

### 15.3 payload 格式要保持兼容

课程索引消息是上下游契约。

字段变更时需要考虑：

- Flink 是否兼容
- ES mapping 是否兼容
- 老消息是否还能被消费

## 16. 总结

当前课程新增、修改、删除的消息可靠发送方案，本质上是 Outbox 本地消息表模式。

核心设计是：

```text
课程业务变更和消息记录同事务提交
事务提交后立即发送 Kafka
发送失败由定时任务兜底重试
多节点通过状态抢占避免并发重复处理
超过最大重试次数进入 DEAD 等待人工处理
```

它解决的核心问题是：

```text
数据库事务和 Kafka 发送无法天然保持一致。
```

通过本地消息表，把“需要发送 Kafka”这件事先可靠地落到数据库里，再通过立即发送和定时补偿，实现课程数据变更到搜索索引同步的最终一致性。
