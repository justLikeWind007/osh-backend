# 工具模块设计文档

## 1. 模块目标

工具模块用于统一管理站内工具资源，覆盖工具的新增、编辑、删除、搜索、推荐、详情查看、收藏、点赞/差评、购买套餐、额度扣减与权限校验。

模块目标：

- 给前端提供统一的工具列表、详情、推荐、购买、使用能力
- 给后台提供统一的工具配置入口
- 支持免费工具、付费工具、会员专属工具、小班专属工具、内部专属工具
- 支持工具使用次数扣减
- 支持通过资源等级控制访问权限
- 支持 ES 搜索索引同步

## 2. 资源类型与权限等级

工具模块中的资源类型与 `level` 一一对应，后端保存时会自动按 `resourceType` 回填 `level`，不依赖前端手工传入。

对应规则如下：

| resourceType | 含义 | level |
| --- | --- | --- |
| `FREE` | 免费 | `1` |
| `CASH_ONLY` | 仅现金购买 | `2` |
| `CASH_POINT` | 现金+积分购买 | `3` |
| `VIP` | VIP专属 | `4` |
| `SMALL_CLASS` | 小班专属 | `5` |
| `INTERNAL` | 内部专属 | `6` |

说明：

- 前端只需要传 `resourceType`
- 后端保存工具时会自动把 `resourceType` 转成对应 `level`
- 后续权限校验统一基于工具表中的 `level`

## 5. 工具新增/编辑设计

### 5.1 请求参数

新增/编辑工具请求对象：

- `ToolSaveRequest`

重要字段：

- `toolName`
- `routePath`
- `description`
- `resourceType`
- `tags`
- `packages`

说明：

- `level` 字段虽然请求对象里还保留，但后端实际保存时不再信任该值
- 最终落库的 `level` 由后端根据 `resourceType` 自动计算

### 5.2 后端保存逻辑

保存逻辑位于：

- `OshToolServiceImpl.buildTool(...)`

处理步骤：

1. 校验访问目标，如 `routePath`
2. 组装 `OshTool`
3. 根据 `resourceType` 自动映射 `level`
4. 保存工具主记录
5. 同步标签
6. 如果是付费类资源，则同步套餐
7. 写入工具索引事件

### 5.3 流程图

```text
[前端提交工具表单]
        |
        v
[OshToolController.save/update]
        |
        v
[OshToolServiceImpl.buildTool]
        |
        +--> 校验 routePath
        |
        +--> 读取 resourceType
        |
        +--> 自动映射 level
        |
        v
[保存 osh_tool]
        |
        +--> 保存标签
        |
        +--> 保存套餐
        |
        +--> 写出索引事件
        |
        v
[返回 toolId]
```

### 5.4 示例

前端提交：

```json
{
  "toolName": "图片转PDF",
  "routePath": "/tool/image-to-pdf",
  "description": "支持多张图片合成为 PDF",
  "resourceType": "VIP",
  "tags": ["PDF工具", "图片工具"]
}
```

后端保存结果：

```json
{
  "resourceType": "VIP",
  "level": 4
}
```

## 6. 工具使用权限设计

### 6.1 权限校验原则

工具使用权限分两层：

1. 资源权限层
2. 使用额度层

先判断用户有没有资格用这个工具，再判断是否需要扣次以及剩余次数是否足够。

### 6.2 资源权限层

工具权限统一通过下面的方法判断：

```java
ResourcePermissionUtil.hasPermission(ResourceTypeEnum.TOOL, toolId)
```

含义：

- 根据资源类型 `TOOL` 找到对应表 `osh_tool`
- 根据 `toolId` 查询工具记录的 `level`
- 对比当前用户等级
- 用户等级满足要求时返回 `true`

### 6.3 使用额度层

如果工具资源类型是以下两类，则还需要额度校验：

- `CASH_ONLY`
- `CASH_POINT`

对应逻辑：

- 查询 `osh_user_tool_quota`
- 判断当前用户对该工具的剩余额度
- 余额大于 0 才允许扣减

如果是：

- `FREE`
- `VIP`
- `SMALL_CLASS`
- `INTERNAL`

则只做权限校验，不做扣次。

### 6.4 使用流程图

```text
[前端请求使用工具]
        |
        v
[调用 /pc/tool/use/check]
        |
        v
[OshToolServiceImpl.checkToolUsagePermission]
        |
        +--> ResourcePermissionUtil.hasPermission(TOOL, toolId)
        |        |
        |        +--> 无权限 -> 返回 useAllowed=false
        |
        +--> 判断 resourceType 是否为付费额度型
                 |
                 +--> 否 -> 允许使用，不扣次
                 |
                 +--> 是 -> 查询剩余额度
                           |
                           +--> 额度不足 -> deductAllowed=false
                           +--> 额度足够 -> deductAllowed=true
```

### 6.5 实际扣次流程

```text
[前端点击正式使用]
        |
        v
[调用 /pc/tool/use/consume]
        |
        v
[先做权限校验]
        |
        +--> 无权限 -> 拒绝
        |
        +--> 有权限
               |
               +--> 免费/会员类工具 -> 直接放行
               |
               +--> 付费额度类工具
                        |
                        +--> 扣减额度
                        |
                        +--> 增加 total_usage
                        |
                        +--> 返回剩余次数
```

### 6.6 示例

示例一：免费工具

- 工具 `resourceType = FREE`
- 工具 `level = 1`
- 用户 `level = 2`

结果：

- 权限通过
- 不需要扣次

示例二：现金购买工具

- 工具 `resourceType = CASH_ONLY`
- 工具 `level = 2`
- 用户 `level = 2`
- 用户剩余额度 `5`

结果：

- 权限通过
- 可以扣减一次
- 返回剩余 `4`

示例三：VIP专属工具

- 工具 `resourceType = VIP`
- 工具 `level = 4`
- 用户 `level = 3`

结果：

- `ResourcePermissionUtil.hasPermission(...) = false`
- 不允许使用

## 7. 工具购买设计

工具购买由 `ToolPurchaseController` 和 `ToolPurchaseServiceImpl` 负责。

主要职责：

- 查询套餐详情
- 创建购买订单
- 支付成功后发放额度

购买后的结果最终落在：

- 购买记录表
- 用户工具额度表

对于 `CASH_ONLY`、`CASH_POINT` 类型工具，用户购买成功后，后续使用时才允许扣减额度。

## 11. 本次实现约束

当前工具模块的关键约束如下：

- 工具保存时以后端 `resourceType -> level` 映射为准
- 前端不应直接控制最终 `level`
- 工具使用必须先做 `ResourcePermissionUtil.hasPermission(ResourceTypeEnum.TOOL, toolId)`
- 只有 `CASH_ONLY` 和 `CASH_POINT` 才进入扣次逻辑
- 其它资源类型只做权限控制，不做额度扣减

## 12. 后续建议

建议后续继续收敛以下几点：

- 将前端资源类型候选项抽成统一枚举配置
- 将 `resourceType -> level` 映射提取为共享枚举，避免多模块各写一套
- 为 `ResourcePermissionUtil` 增加更好的 Spring 单测支持
- 为工具购买、扣次、权限判断补更多集成测试
