# 工具模块订单超时处理设计

## 1. 背景

当前工具模块已经接入统一支付链路，支持：

- 工具购买下单
- 纯现金支付
- 现金 + 积分支付
- 支付成功后发放工具使用次数

在支付能力接通后，工具订单还需要补齐“超时未支付自动关闭”的处理，否则会出现以下问题：

- 待支付订单长期挂起
- 现金 + 积分支付场景下，积分已扣但订单未支付成功时无法自动退回
- 用户页面轮询时会一直看到待支付状态
- 后续重复下单和额度统计会产生歧义

因此当前实现补充了工具订单超时关闭能力。

## 2. 当前实现目标

当前工具订单超时处理的目标不是依赖第三方支付回调，而是基于本地支付流水主动扫描和主动关单。

核心目标：

- 统一在本地判定哪些订单已经超时
- 每分钟自动扫描一次超时待支付订单
- 超时后关闭支付单、订单主单、工具购买记录
- 若为现金 + 积分订单，则自动退回积分
- 关闭后前端轮询能感知订单已关闭

## 3. 设计原则

### 3.1 不依赖第三方回调

当前支付链路设计上不依赖第三方支付回调作为最终一致性来源。

原因：

- 本地开发环境下第三方回调地址不稳定
- 部分测试场景不希望依赖外网回调
- 当前前端本身已经具备轮询支付状态能力

所以订单超时关闭采用：

- 本地写入支付过期时间
- Spring 定时任务主动扫描
- 本地调用统一关单逻辑

### 3.2 超时处理统一走 `cancelPayment`

无论是：

- 用户主动手动关闭订单
- 系统定时任务自动关闭超时订单

都统一调用：

- `UnifiedOrderService.cancelPayment(paymentNo)`

这样可以保证工具购买单、积分退款、订单状态更新都走同一条路径，避免逻辑分叉。

## 4. 核心数据

当前超时处理依赖以下数据：

### 4.1 `osh_payment`

关键字段：

- `payment_no`
- `status`
  - `0`：待支付
  - `1`：支付成功
  - `2`：支付失败
  - `3`：已关闭
- `expire_time`

作用：

- 作为超时扫描的主表
- 只扫描 `status = 0` 且 `expire_time <= now()` 的记录

### 4.2 `osh_order`

关键字段：

- `order_no`
- `status`
  - `0`：待支付
  - `1`：已支付
  - `2`：已关闭/已取消
- `close_time`

作用：

- 与支付流水状态保持一致
- 超时关闭后同步更新主订单状态

### 4.3 `osh_tool_purchase_record`

关键字段：

- `order_no`
- `payment_no`
- `package_pay_type_snapshot`
- `package_point_amount_snapshot`
- `order_status`
- `grant_status`

作用：

- 保存工具购买事实
- 超时关闭时同步更新工具购买记录状态
- 现金 + 积分订单关闭时据此判断是否需要退积分

## 5. 支付过期时间写入

当前在统一支付下单时，会给支付流水写入过期时间。

实现位置：

- `UnifiedOrderServiceImpl.buildPendingPayment(...)`

处理方式：

- 创建 `osh_payment` 时写入：
  - `expire_time = LocalDateTime.now().plusMinutes(pay.expire-minutes)`

默认超时时间：

- `30` 分钟

配置项：

- `pay.expire-minutes`

若未配置，则默认按 30 分钟处理。

## 6. 定时扫描任务

### 6.1 实现方式

当前采用 Spring `@Scheduled` 定时任务，而不是后台动态 Quartz 配置任务。

原因：

- 需求固定明确：每分钟扫描一次
- 项目主应用已启用 `@EnableScheduling`
- 实现简单，利于稳定落地

### 6.2 任务入口

实现类：

- `com.backstage.system.service.order.impl.OrderCloseTask`

调度配置：

```java
@Scheduled(cron = "0 * * * * ?")
```

含义：

- 每分钟第 0 秒执行一次

### 6.3 扫描条件

扫描 SQL 对应逻辑：

- 查询 `osh_payment`
- 条件：
  - `status = 0`
  - `delete_flag = 0`
  - `expire_time is not null`
  - `expire_time <= 当前时间`

也就是说，当前只会关闭：

- 已下单
- 仍待支付
- 已达到超时时间

的支付流水。

## 7. 超时关闭流程

### 7.1 定时任务层

定时任务每分钟执行一次：

1. 查询所有超时待支付流水
2. 遍历每一条流水
3. 调用：
   - `unifiedOrderService.cancelPayment(paymentNo)`

### 7.2 统一支付层

`UnifiedOrderService.cancelPayment(paymentNo)` 当前会执行以下操作：

1. 查询支付流水
2. 校验必须是待支付状态
3. 更新 `osh_payment.status = 3`
4. 更新 `osh_order.status = 2`
5. 写入 `close_time`
6. 调用工具购买业务的取消逻辑：
   - `toolPurchaseService.cancelPendingPurchase(paymentNo)`

### 7.3 工具购买业务层

`ToolPurchaseService.cancelPendingPurchase(paymentNo)` 会执行：

1. 根据 `payment_no` 查询 `osh_tool_purchase_record`
2. 若购买记录不存在，直接返回
3. 若已是关闭状态，直接返回
4. 若已发放成功，直接返回
5. 若支付类型为现金 + 积分，且积分快照大于 0：
   - 退回积分
   - 记录积分流水
   - 刷新 Redis 用户资产缓存
6. 更新 `osh_tool_purchase_record.order_status = 2`

## 8. 现金 + 积分订单的特殊处理

工具购买里，现金 + 积分场景在创建订单时会先扣减积分。

因此若订单后续超时未支付，必须把积分退回，否则用户资产会被错误占用。

当前实现里：

- 下单成功时：
  - 扣减 `osh_user_asset.points`
  - 写入 `osh_user_asset_record`
- 订单关闭时：
  - 根据 `osh_tool_purchase_record.package_point_amount_snapshot` 退回积分
  - 再写一条返还积分流水

也就是说，工具超时关闭不仅关闭订单，还会做积分逆向补偿。

## 9. 幂等处理

当前超时关闭具备基本幂等性，体现在以下几层：

### 9.1 `osh_payment`

`cancelPayment(paymentNo)` 只允许关闭：

- `status = 0` 的支付单

若支付已成功或已关闭，不会重复进入关闭逻辑。

### 9.2 `osh_tool_purchase_record`

在工具购买取消逻辑中，若满足以下任一条件会直接返回：

- 购买记录不存在
- `order_status = 2`
- `grant_status = 1`

这样可以避免：

- 重复更新购买记录
- 重复退积分
- 已发放成功后又被错误关闭

## 10. 前端联动效果

当前前端工具详情页已接入支付状态轮询。

当超时关闭发生后：

- `GET /pc/pay/status`
  会返回订单已关闭/支付已关闭状态
- 前端轮询识别到：
  - `paymentStatus = 3`
  - 或 `orderStatus = 2`

则会：

- 停止轮询
- 更新前端支付状态
- 支持弹窗关闭

因此用户能感知订单已自动失效。

## 11. 当前方案的优点

### 11.1 实现简单

- 无需等待第三方回调
- 无需额外的支付状态补偿平台
- 直接用本地数据判断超时

### 11.2 对工具购买场景闭环完整

- 关闭支付流水
- 关闭主订单
- 关闭工具购买记录
- 退回积分

### 11.3 可复用现有手动关单逻辑

当前自动关单不是另起一套逻辑，而是复用：

- `UnifiedOrderService.cancelPayment(paymentNo)`

这能确保手动关单和自动关单行为一致。

## 12. 当前限制

### 12.1 不主动向第三方发起“远端关单”

当前实现只更新本地状态，不额外调用第三方支付平台的关闭接口。

也就是说：

- 本地认为订单已关闭
- 第三方平台那边若仍保留支付链接，不影响本地结果

当前业务前提是：

- 后续统一通过主动查询支付状态为准
- 不依赖第三方回调纠偏

### 12.2 依赖本地过期时间正确写入

如果支付流水创建时未写 `expire_time`，则该支付单不会被自动扫描到。

当前已经补齐这部分逻辑，但后续若有新的下单链路接入统一支付，也要确保复用同样的写入方式。

## 13. 建议的后续增强

虽然当前实现已经可用，但后续可以继续增强：

### 13.1 增加关闭原因字段

可在 `osh_tool_purchase_record` 或 `osh_order` 增加：

- `close_reason`

如：

- `USER_CANCEL`
- `TIMEOUT_CLOSE`

当前实现里未区分关闭来源，只是统一更新为关闭状态。

### 13.2 增加关单日志

若后续要做支付排障和客服查询，建议补充订单关闭操作日志。

### 13.3 对自动关单增加批量限制

如果后续待支付单量很大，可以考虑：

- 分批扫描
- 分页处理
- 防止单次任务处理过多记录

当前阶段使用简单全量扫描，足够支撑现有规模。

## 14. 结论

当前工具模块订单超时处理已经按“本地主动扫描 + 统一关单 + 工具业务补偿”的方式落地。

实际处理链路为：

1. 下单时写入 `osh_payment.expire_time`
2. Spring 定时任务每分钟扫描超时待支付单
3. 调用统一支付关闭逻辑
4. 同步关闭主订单与工具购买记录
5. 若为现金 + 积分订单，则退回积分
6. 前端轮询感知关闭状态并结束支付流程

这套方案不依赖第三方回调，已经满足当前工具购买场景下的超时关闭需求。
