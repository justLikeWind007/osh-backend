# 接入支付模块说明

 > 更新时间：2026-05-31

---

## 一、接入目标

现在支付模块已经提供了统一下单、支付流水、支付回调、支付成功消息和前端轮询能力。

其他业务模块后续如果需要售卖课程、电子书、工具、会员、专栏等商品，不需要自己再写一套支付逻辑，只需要做三件事：

1. 业务模块自己校验商品、价格、库存、购买资格
2. 调用统一支付模块创建订单和支付流水
3. 实现支付成功后的业务发放逻辑

支付模块负责：

- 创建 `osh_order`
- 创建 `osh_payment`
- 调用易支付生成二维码或支付链接
- 接收支付平台回调
- 校验签名和金额
- 推进订单、支付流水状态
- 发送支付成功消息
- 根据商品类型路由到业务模块的支付成功处理器

---

## 二、核心原则

### 2.1 业务模块是购买入口

前端不要直接调用支付模块创建订单。

正确流程是：

```text
前端
  ↓
业务模块结算接口
  ↓
业务模块校验商品和价格
  ↓
OrderCheckoutService.checkout(...)
  ↓
统一支付模块创建订单和支付流水
```

例如：

```text
买电子书：POST /api/book/checkout
买工具套餐：POST /pc/tool/purchase
买课程：POST /api/course/checkout
```

业务模块对外暴露自己的购买或结算接口，支付模块只作为后端内部能力被调用。

### 2.2 支付成功后的权益发放由业务模块负责

支付模块只知道“某个订单已经支付成功”，但不知道具体业务应该怎么发放权益。

所以每个业务模块都需要实现自己的 `OrderPaidHandler`：

```text
电子书支付成功 → 开通电子书阅读权限
工具套餐支付成功 → 增加工具使用次数
课程支付成功 → 开通课程学习权限
会员支付成功 → 延长会员有效期
```

## 三、当前支付模块提供的接入点

### 3.1 统一下单入口

业务模块调用：

```java
com.backstage.system.service.order.OrderCheckoutService
```

核心方法：

```java
OrderCheckoutRespVO checkout(OrderCheckoutReqVO reqVO);
```

该方法会完成：

1. 校验结算参数
2. 生成业务订单号 `orderNo`
3. 生成支付流水号 `paymentNo`
4. 写入 `osh_order`
5. 写入 `osh_payment`
6. 免费订单直接完成支付
7. 非免费订单调用支付平台生成二维码或支付链接
8. 返回订单号、支付流水号和支付参数

### 3.2 请求参数

`OrderCheckoutReqVO` 是业务模块传给支付模块的结算参数。

| 字段 | 必填 | 说明 |
|------|------|------|
| `userId` | 是 | 下单用户 ID |
| `productType` | 是 | 商品类型，对应 `ProductTypeEnum` 的 code |
| `productId` | 是 | 商品 ID |
| `productName` | 是 | 商品名称快照 |
| `purchaseMode` | 否 | 购买方式，普通购买可为空 |
| `activityId` | 否 | 活动 ID，秒杀、拼团等场景使用 |
| `originalAmount` | 是 | 商品原价 |
| `discountAmount` | 否 | 优惠金额 |
| `payableAmount` | 是 | 实付金额 |
| `couponId` | 否 | 优惠券 ID |
| `channel` | 否 | 支付渠道，如 `wxpay`、`alipay` |
| `usePoints` | 否 | 是否使用积分抵扣。仅 `true` 时扣减积分；不传、`null`、`false` 都视为不使用积分 |
| `clientIp` | 否 | 客户端 IP。HTTP 场景可不传，支付模块会尝试自动获取；异步或非 HTTP 场景建议传固定标识 |

当前已定义的商品类型：

| code | name | 说明 |
|------|------|------|
| `1` | `course` | 课程 |
| `2` | `book` | 电子书 |
| `3` | `column` | 专栏 |
| `4` | `seckill` | 秒杀 |
| `5` | `tool` | 工具 |

当前已支持的支付渠道：

| channel | 说明 |
|---------|------|
| `wxpay` | 微信支付 |
| `alipay` | 支付宝 |
| `free` | 免费订单，通常由支付模块根据金额自动处理 |

如果 `channel` 为空或无法识别，当前实现默认走 `wxpay`。

### 3.3 积分抵扣参数说明

支付模块已支持结算时使用积分抵扣，当前兑换比例为：

```text
10 积分 = 1 元
```

业务模块接入时只需要关注 `usePoints`：

| 场景 | 传参 | 支付模块行为 |
|------|------|--------------|
| 老业务暂未适配积分抵扣 | 不传 `usePoints` | 默认不抵扣积分，保持原现金支付逻辑 |
| 用户不使用积分 | `usePoints=false` | 不抵扣积分 |
| 用户选择使用积分 | `usePoints=true` | 按用户可用积分和订单金额自动计算可抵扣金额 |

兼容性要求：

- `usePoints` 是可选字段，老业务模块不传不会报错
- 只有 `Boolean.TRUE.equals(usePoints)` 时才会扣减积分
- 积分不足时不会直接失败，会用可用积分尽量抵扣，剩余部分继续走现金支付
- 积分足够覆盖整单时，`needPay=false`，支付模块会直接完成订单
- 积分扣减和订单、支付流水创建在同一事务内完成
- 使用积分抵扣时会写入用户资产流水，用于后续审计和问题排查

示例：

```java
OrderCheckoutReqVO orderReqVO = new OrderCheckoutReqVO();
orderReqVO.setUserId(userId);
orderReqVO.setProductType(ProductTypeEnum.BOOK.getCode());
orderReqVO.setProductId(bookId);
orderReqVO.setProductName(bookName);
orderReqVO.setOriginalAmount(bookPrice);
orderReqVO.setPayableAmount(bookPrice);
orderReqVO.setUsePoints(reqVO.getUsePoints()); // 可选，不传则默认不抵扣
```

### 3.4 响应参数

`OrderCheckoutRespVO` 是支付模块返回给业务模块的结果。

| 字段 | 说明 |
|------|------|
| `needPay` | 是否需要前端继续支付 |
| `orderNo` | 业务订单号 |
| `paymentNo` | 支付流水号 |
| `payStatus` | 当前支付状态 |
| `price` | 本次还需要现金支付的金额。使用积分后这里是抵扣后的剩余现金金额 |
| `pointsUsed` | 本单实际使用积分数量。未使用积分时为 `0` |
| `pointsDeductAmount` | 本单积分抵扣金额，单位元。未使用积分时为 `0.00` |
| `remainingPoints` | 抵扣后剩余积分。未使用积分时可能为空 |
| `expireTime` | 支付过期时间 |
| `closeExpireMinutes` | 支付关闭倒计时分钟数 |
| `payment` | 支付渠道参数，包含二维码和支付链接 |

需要支付时，业务模块一般把 `orderNo`、`paymentNo`、`price`、`pointsUsed`、`pointsDeductAmount`、`remainingPoints`、`payment.qrcode`、`payment.payUrl` 返回给前端。

免费订单时，`needPay = false`，支付模块会直接推进订单为已支付，并发送支付成功消息。

前端展示建议：

| 返回值组合 | 展示方式 |
|------------|----------|
| `pointsUsed=0` 且 `needPay=true` | 展示正常现金支付金额 `price` |
| `pointsUsed>0` 且 `needPay=true` | 展示“已用积分抵扣 ¥pointsDeductAmount，还需支付 ¥price” |
| `pointsUsed>0` 且 `needPay=false` | 展示“积分已全额抵扣，本单无需继续支付” |
| `pointsUsed=0` 且 `needPay=false` | 展示免费订单完成 |

## 四、业务模块接入步骤

### 4.1 第一步：准备业务结算接口

每个业务模块对外提供自己的结算接口。

示例：

```java
@PostMapping("/checkout")
public R<OrderCheckoutRespVO> checkout(@RequestBody XxxCheckoutReqVO reqVO) {
    // 1. 校验商品是否存在、是否上架、是否允许购买
    // 2. 计算原价、优惠金额、实付金额
    // 3. 如有业务侧购买记录，先写入待支付记录
    // 4. 调用 OrderCheckoutService.checkout(...)
    // 5. 保存 orderNo、paymentNo 到业务侧记录
    return R.ok(orderCheckoutService.checkout(orderReqVO));
}
```

业务模块需要自己负责：

- 商品是否存在
- 商品是否可购买
- 用户是否已经购买过
- 库存、次数、套餐是否有效
- 原价和实付金额是否正确
- 业务侧待支付记录是否需要落库

支付模块不处理这些业务规则。

### 4.2 第二步：组装 `OrderCheckoutReqVO`

示例：

```java
OrderCheckoutReqVO orderReqVO = new OrderCheckoutReqVO();
orderReqVO.setUserId(userId);
orderReqVO.setProductType(ProductTypeEnum.TOOL.getCode());
orderReqVO.setProductId(toolId);
orderReqVO.setProductName(toolName);
orderReqVO.setOriginalAmount(originalAmount);
orderReqVO.setDiscountAmount(discountAmount);
orderReqVO.setPayableAmount(payableAmount);
orderReqVO.setChannel(reqVO.getChannel());
orderReqVO.setUsePoints(reqVO.getUsePoints()); // 可选：老业务不传时默认不抵扣积分

OrderCheckoutRespVO respVO = orderCheckoutService.checkout(orderReqVO);
```

注意：

- `productType` 必须使用 `ProductTypeEnum` 中已定义的类型
- `productName` 是订单快照，不要只传 ID
- 金额统一使用 `BigDecimal`
- `payableAmount = 0` 时会走免费订单逻辑
- `usePoints` 可不传；不传、`null`、`false` 均视为不使用积分
- `usePoints = true` 时，支付模块会自动按 `10 积分 = 1 元` 计算抵扣金额
- 积分不足时，支付模块会扣完可用积分，剩余现金金额通过 `respVO.price` 返回给前端继续支付
- 业务模块不要自己生成 `orderNo` 和 `paymentNo`，由支付模块统一生成

### 4.3 第三步：实现支付成功处理器

业务模块需要实现：

```java
com.backstage.system.service.order.OrderPaidHandler
```

示例：

```java
@Component
public class XxxPaidHandler implements OrderPaidHandler {

    @Override
    public String bizType() {
        return ProductTypeEnum.TOOL.getName();
    }

    @Override
    @Transactional(rollbackFor = Exception.class)
    public void handle(String orderNo) {
        // 1. 根据 orderNo 查询业务侧购买记录
        // 2. 判断是否已经发放，保证幂等
        // 3. 发放权益
        // 4. 更新业务侧订单状态和发放状态
    }
}
```

`bizType()` 返回值必须和 `ProductTypeEnum.getName()` 一致。

支付成功后，支付模块会发送 Kafka 消息，`PaymentSuccessConsumer` 会根据订单中的 `productType` 找到对应的 `OrderPaidHandler` 并执行 `handle(orderNo)`。

### 4.4 第四步：保证支付成功处理幂等

支付平台可能重复回调，Kafka 消息也可能重复消费。

所以业务模块的 `OrderPaidHandler.handle(orderNo)` 必须能重复执行。

推荐做法：

```text
1. 根据 orderNo 查询业务侧记录
2. 如果已经发放成功，直接 return
3. 如果还未发放，执行权益发放
4. 发放成功后更新 grant_status / order_status
5. 发放失败时记录失败原因，并抛出异常或记录日志
```

当前工具模块可以参考：

```text
ToolPaidHandler
```

电子书模块可以参考：

```text
BookPaidHandler
```

## 五、前端如何配合

### 5.1 下单

前端调用业务模块自己的结算接口。

业务模块返回：

```json
{
  "needPay": true,
  "orderNo": "tool20260520123456",
  "paymentNo": "P20260520123456",
  "payStatus": "0",
  "price": 7.90,
  "pointsUsed": 20,
  "pointsDeductAmount": 2.00,
  "remainingPoints": 180,
  "payment": {
    "channel": "wxpay",
    "qrcode": "weixin://wxpay/xxx",
    "payUrl": "https://xxx"
  }
}
```

如果 `needPay = true`，前端展示二维码或跳转支付链接。

如果 `needPay = false`，说明免费订单已经完成，前端可以直接进入业务成功页。

如果返回了积分抵扣信息，前端需要同时展示：

```text
商品金额：原业务模块展示自己的商品价格
积分抵扣：pointsUsed 积分，抵扣 pointsDeductAmount 元
还需支付：price 元
剩余积分：remainingPoints
```

当用户积分不足以覆盖整单时，`price` 就是还需要现金支付的金额，`needPay=true`。前端继续展示二维码或支付链接即可。

当用户积分足够覆盖整单时，`price=0` 且 `needPay=false`。前端不要再拉起支付，直接展示购买成功或等待业务权益开通结果。

电子书模块当前的购买弹窗位于：

```text
pages/detail/[type]/[id].vue
```

它通过 `UserBookRelationController.purchase` 创建电子书订单，再把 `orderNo`、`paymentNo` 和二维码信息交给弹窗展示。

### 5.2 轮询支付状态

前端拿到 `orderNo` 后，统一按业务订单号轮询：

```text
GET /pc/pay/status?orderNo={orderNo}
```

这里建议前端按 `orderNo` 查，而不是按 `paymentNo` 查。

原因是当前实现中，订单和支付流水始终是一对一关系：

```text
一次 checkout -> 一条 osh_order -> 一条 osh_payment
```

前端关心的是“这个业务订单是否支付成功”，所以按 `orderNo` 轮询更符合业务语义。

支付模块内部仍然会根据 `orderNo` 找到关联的 `paymentNo`。如果本地支付流水仍是待支付状态，后端会用 `paymentNo` 主动查询支付平台，并在确认已支付后推进 `osh_payment` 和 `osh_order` 状态。

`paymentNo` 仍然用于支付平台回调、取消支付、超时关闭和支付排障，不作为前端轮询的主参数。

返回结果中重点看：

| 字段 | 说明 |
|------|------|
| `orderStatus` | 订单状态 |
| `paymentStatus` | 支付流水状态 |
| `payStatus` | 是否已支付 |

轮询只用于用户体验。

真正的订单状态推进和权益发放，仍以后端支付回调和支付成功消息为准。

前端实现上，电子书详情页会复用 `composables/order.js` 里的统一封装：

```js
usePayStatusApi(orderNo)
useCancelPayApi(orderNo)
```

这两个封装都通过 `params: { orderNo }` 调用 `/pc/pay/status` 和 `/pc/pay/cancel`，和后端 `PayController` 的 `@RequestParam String orderNo` 保持一致。

## 六、完整流程

```text
用户点击购买
  ↓
前端调用业务模块结算接口
  ↓
业务模块校验商品、价格、购买资格
  ↓
业务模块调用 OrderCheckoutService.checkout(...)
  ↓
支付模块写入 osh_order
  ↓
支付模块写入 osh_payment
  ↓
支付模块调用易支付生成二维码
  ↓
业务模块把 orderNo、paymentNo、二维码返回给前端
  ↓
前端展示二维码并调用 /pc/pay/status 按 orderNo 轮询支付状态
  ↓
用户完成支付
  ↓
易支付回调 /notify/pay
  ↓
支付模块验签、校验金额、更新订单和支付流水
  ↓
支付模块发送支付成功 Kafka 消息
  ↓
PaymentSuccessConsumer 根据 productType 找到 OrderPaidHandler
  ↓
业务模块发放权益
```

## 七、新业务类型接入清单

如果后续新增一个商品类型，例如会员，需要按下面顺序处理。

### 7.1 增加商品类型枚举

在 `ProductTypeEnum` 中增加类型：

```java
MEMBER(6, "member", "会员")
```

注意：

- `code` 不能和已有类型重复
- `name` 是支付成功处理器路由用的业务标识
- `desc` 用于展示和排查

### 7.2 新增业务结算接口

例如：

```text
POST /api/member/checkout
```

该接口内部完成会员套餐校验、价格计算，然后调用 `OrderCheckoutService.checkout(...)`。

### 7.3 新增支付成功处理器

```java
@Component
public class MemberPaidHandler implements OrderPaidHandler {

    @Override
    public String bizType() {
        return ProductTypeEnum.MEMBER.getName();
    }

    @Override
    public void handle(String orderNo) {
        // 根据 orderNo 查询会员购买记录
        // 幂等校验
        // 开通或延长会员
        // 更新发放状态
    }
}
```

启动时 `OrderPaidHandlerRegistry` 会自动收集所有 `OrderPaidHandler`。

如果 `bizType()` 不在 `ProductTypeEnum` 中，系统会直接抛错，避免支付成功后找不到业务处理逻辑。

### 7.4 补充业务侧订单状态处理

业务模块如果有自己的购买记录表，建议至少保留这些字段：

| 字段 | 说明 |
|------|------|
| `user_id` | 用户 ID |
| `product_id` | 商品 ID |
| `order_no` | 统一订单号 |
| `payment_no` | 支付流水号 |
| `order_status` | 业务侧订单状态 |
| `grant_status` | 权益发放状态 |
| `grant_time` | 权益发放时间 |
| `grant_fail_reason` | 发放失败原因 |

业务侧状态必须支持重复处理，不能因为重复回调或重复消息导致重复发放。

## 八、注意事项

1. 业务模块只调用 `OrderCheckoutService.checkout(...)`，不要直接操作 `osh_order`、`osh_payment`
2. 业务模块必须先校验自己的商品和价格，再调用支付模块
3. 金额必须使用 `BigDecimal`，禁止用 `double` 或 `float`
4. 支付成功处理器必须做幂等校验
5. `productType` 必须和 `ProductTypeEnum` 保持一致
6. `bizType()` 必须返回 `ProductTypeEnum.getName()`
7. 免费订单也会进入支付成功后置流程，业务模块不能只考虑扫码支付场景
8. 前端轮询只是体验手段，不能代替后端支付回调
9. 取消支付或支付超时应统一调用 `OrderService.cancelPaymentByOrderNo(orderNo)`
10. `usePoints` 是可选参数，老业务模块可以暂时不传；不传时默认不抵扣积分
11. 如果业务模块支持积分抵扣，需要把 `pointsUsed`、`pointsDeductAmount`、`remainingPoints`、`price`、`needPay` 原样返回给前端
12. 新业务接入后，至少要验证成功支付、免费订单、重复回调、金额异常、支付超时、积分不使用、积分部分抵扣、积分全额抵扣这几类场景

## 九、最小接入示例

```java
@Service
public class XxxPurchaseService {

    @Resource
    private OrderCheckoutService orderCheckoutService;

    /**
     * 创建业务订单并发起支付。
     */
    public OrderCheckoutRespVO checkout(Long userId, Long productId, String channel) {
        // 查询和校验商品
        XxxProduct product = getAvailableProduct(productId);

        // 组装统一支付结算参数
        OrderCheckoutReqVO orderReqVO = new OrderCheckoutReqVO();
        orderReqVO.setUserId(userId);
        orderReqVO.setProductType(ProductTypeEnum.TOOL.getCode());
        orderReqVO.setProductId(product.getId());
        orderReqVO.setProductName(product.getName());
        orderReqVO.setOriginalAmount(product.getPrice());
        orderReqVO.setPayableAmount(product.getPrice());
        orderReqVO.setChannel(channel);

        // orderReqVO.setUsePoints(usePoints);

        return orderCheckoutService.checkout(orderReqVO);
    }
}
```
