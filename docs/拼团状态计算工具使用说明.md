# 拼团状态计算工具使用说明

## 背景

拼团模块采用**方案一（查询时状态映射）**，为了避免每次都要重复写状态映射逻辑，封装了统一的计算工具。

## 核心方法

### `calculateGroupStatus()`

**位置**: `OshGroupServerServiceImpl.java`

**签名**:
```java
private GroupStatusResult calculateGroupStatus(
    Integer groupStatus,      // 数据库中的原始状态
    Integer currentNum,       // 当前人数
    Integer maxNum,           // 最大人数
    LocalDateTime serverExpireTime  // 服务器到期时间
)
```

**返回**: `GroupStatusResult` 对象，包含：
- `status`: 前端展示状态（1=进行中, 2=拼团成功, 3=已结束）
- `groupStatus`: 数据库原始状态（0/1/2/3）
- `isFull`: 是否人数已满
- `isTimeInsufficient`: 是否服务器剩余时间不足30天
- `canJoin()`: 方法，判断是否可以参团

---

## 使用场景

### 场景1: 详情查询（最常用）

```java
// 在 buildDetailFromUserInitiated 方法中
GroupStatusResult statusResult = calculateGroupStatus(
        initiated.getGroupStatus(),
        initiated.getCurrentNum(),
        initiated.getMaxNum(),
        initiated.getServerExpireTime()
);

vo.setStatus(statusResult.status);
vo.setGroupStatus(statusResult.groupStatus);
vo.setGroupStatusText(getGroupStatusText(statusResult.groupStatus));
vo.setCanJoin(statusResult.canJoin());
```

**优势**: 
- 一行代码完成所有状态计算
- 自动处理人数上限和时间检查
- 保证逻辑一致性

---

### 场景2: 列表查询

```java
// 在 selectGroupActivityList 方法中
for (GroupActivityListVO vo : list) {
    // 列表查询没有 group_status 和 serverExpireTime，传 null
    GroupStatusResult statusResult = calculateGroupStatus(
            null,  
            vo.getCurrentNum(),
            vo.getGroupMaxNum(),
            null   
    );
    
    vo.setCanJoin(statusResult.canJoin());
}
```

**注意**: 
- 列表查询的 `group_status` 可能在 XML 中已映射为 `status`
- 如果传 `null`，会使用默认值（status=1, groupStatus=0）

---

### 场景3: 参团校验

```java
// 在 joinGroup 方法中
GroupStatusResult statusResult = calculateGroupStatus(
        initiated.getGroupStatus(),
        initiated.getCurrentNum(),
        initiated.getMaxNum(),
        initiated.getServerExpireTime()
);

// 检查是否可以参团
if (!statusResult.canJoin()) {
    throw new ServiceException("该拼团已结束");
}

// 检查人数上限
if (statusResult.isFull) {
    throw new ServiceException("该拼团人数已满");
}

// 检查服务器剩余时间
if (statusResult.isTimeInsufficient) {
    throw new ServiceException("服务器剩余使用时间不足一个月");
}
```

**优势**:
- 不用再手动写3个判断条件
- 自动保持与查询逻辑一致

## 状态码定义

### 数据库字段 `group_status`

| 值 | 含义 | 说明 |
|---|------|------|
| 0 | 进行中（未成团） | 当前人数 < 最低成团人数 |
| 1 | 拼团成功 | 已达到最低人数，可继续加入 |
| 2 | 拼团完成 | 同状态1，用于区分业务场景 |
| 3 | 拼团结束 | 人数已满 OR 时间不足30天 |

### 前端字段 `status`（计算得出）

| 值 | 含义 | 映射逻辑 |
|---|------|---------|
| 1 | 进行中 | group_status = 0 |
| 2 | 拼团成功 | group_status = 1 或 2 |
| 3 | 已结束 | group_status = 3 OR 人数满 OR 时间不足 |

---

## 状态流转图

```
[状态0: 进行中]
    ↓ 支付成功，达到最低人数
[状态1: 拼团成功] ←→ [状态2: 拼团完成]
    ↓ 支付成功，达到人数上限
    ↓ OR 服务器剩余时间 < 30天
[状态3: 拼团结束] (终态)
```

## 扩展新状态

如果需要添加新状态（如"即将结束"），只需修改 `calculateGroupStatus()` 方法：

```java
// 示例：添加状态4（即将结束）
if (isTimeInsufficient && !isFull) {
    result.status = 4;  // 新增状态
    result.groupStatus = groupStatus;
    return result;
}
```

**所有调用处自动生效**，无需修改其他代码。

## 常见问题

### Q1: 为什么列表查询传 null？

**A**: 列表查询的 SQL 中已经通过 `CASE WHEN` 映射了 `status`，Java 层不需要再次计算。但为了计算 `canJoin`，仍然调用工具方法。

### Q2: 如果 `groupStatus` 为 null 会怎样？

**A**: 会使用默认值（status=1, groupStatus=0），表示"进行中"。

### Q3: 为什么不在数据库层面维护 status？

**A**: 
1. **数据一致性**: 查询时实时计算，永远是最新状态
2. **并发安全**: 不需要在每次人数变动时更新状态
3. **可追溯性**: `group_status` 保留真实业务节点
4. **灵活性**: 修改映射逻辑不需要改数据库

---

## 最佳实践

1. **统一入口**: 所有状态判断都使用 `calculateGroupStatus()`
2. **不要硬编码**: 避免 `if (status == 1 || status == 2)` 这样的写法
3. **使用 canJoin()**: 判断是否可参团时，优先使用封装方法
4. **记录变更原因**: 支付回调中记录状态变更的原因（人数满/时间不足）
