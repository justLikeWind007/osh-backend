# 实用网站模块接口文档

## 1. 文档说明

- 文档范围：基于当前仓库代码整理，覆盖实用网站模块的公开端、用户交互、管理审核、首页热门、ES 运维接口。
- 代码基准：
  - `backstage-system/src/main/java/com/backstage/system/controller/website`
  - `backstage-system/src/main/java/com/backstage/system/controller/homepage/OshHomePageWebsiteController.java`
  - `backstage-system/src/main/java/com/backstage/system/service/website/impl`
- 整理日期：2026-06-02
- 说明：本文档以“当前代码真实行为”为准，包含接口路径、参数、响应结构、权限、状态枚举、审核流、ES 降级逻辑以及代码中的一些实现细节。

## 2. 模块总览

实用网站模块当前对外接口分为 5 组：

1. 实用网站公开接口：`/pc/website`
2. 用户交互接口：提交、收藏、取消收藏、我的收藏、评价
3. 管理审核接口：待审核列表、审核详情、审核操作、批量删除
4. 首页热门网站接口：`/pc/homepage/website`
5. ES 运维接口：`/pc/website/es`

## 3. 基础约定

### 3.1 认证方式

#### 3.1.1 公开接口

带 `@Anonymous` 的接口可匿名访问，当前包括：

- `GET /pc/website/notices`
- `GET /pc/website/dynamics`
- `GET /pc/website/tags`
- `POST /pc/website/list`
- `PUT /pc/website/click`
- `GET /pc/homepage/website/hot`
- `POST /pc/website/es/recreateIndex`
- `POST /pc/website/es/syncAll`

#### 3.1.2 登录接口

需要登录的接口依赖请求头 `token`。

未登录或 token 失效时，统一认证层通常返回：

```json
{
  "code": 401,
  "msg": "未登录或Token已过期"
}
```

#### 3.1.3 权限点

当前模块涉及的权限包括：

| 权限值 | 说明 |
| --- | --- |
| `website:submit` | 提交网站 |
| `website:favorite` | 收藏网站 |
| `website:favorite:cancel` | 取消收藏 |
| `website:favorite:list` | 我的收藏列表 |
| `website:audit` | 审核网站 |
| `website:audit:list` | 查询待审核列表 |
| `website:audit:detail` | 查询待审核详情 |
| `website:delete:batch` | 批量删除网站 |
| `website:rating:submit` | 提交网站评价 |

### 3.2 请求头

| Header | 必填 | 说明 |
| --- | --- | --- |
| `token` | 登录接口必填 | 用户登录态令牌 |
| `Content-Type: application/json` | Body 为 JSON 时必填 | POST/PUT 带 JSON 使用 |

### 3.3 公共响应格式

#### 3.3.1 `R<T>` 响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | Integer | 200 成功，500 业务失败，401 未登录 |
| `msg` | String | 响应消息 |
| `data` | Object/Array/null | 实际业务数据 |

#### 3.3.2 `TableDataInfo` 分页响应

```json
{
  "total": 15,
  "rows": [],
  "code": 200,
  "msg": "查询成功"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `total` | Long | 总记录数 |
| `rows` | Array | 当前页数据 |
| `code` | Integer | 状态码 |
| `msg` | String | 响应消息 |

#### 3.3.3 网站列表接口的特殊分页格式

`POST /pc/website/list` 没有直接返回 `TableDataInfo`，而是包装成：

```json

  "code": 200,
  "msg": "ok",
  "data": {
    "rows": [],
    "total": 15,
    "pageNum": 1,
    "pageSize": 10
  }

```

#### 3.3.4 我的收藏列表接口的特殊格式

`GET /pc/website/Favorites` 返回：

```json

  "msg": "操作成功",
  "data": {
    "total": 3,
    "rows": []
  }
}
```

## 5. 核心业务规则

### 5.1 网站列表查询优先走 ES，失败后降级 MySQL

`POST /pc/website/list` 的真实流程：

1. 优先查 ES
2. 如果 ES 有结果，则直接使用 ES 返回的分页结果
3. 然后再回填 MySQL 实时计数：
   - `goodCount`
   - `midCount`
   - `badCount`
4. 如果当前请求带有效登录态，还会补 `myRatingType`
5. 如果 ES 查不到或异常，则降级走 MySQL

### 5.2 网站列表中的 `myRatingType`

虽然网站列表接口是匿名可访问的，但如果请求里携带了有效 `token`，系统会软识别当前用户，并额外回填每个网站的：

- `myRatingType`

含义：

| 值 | 含义 |
| --- | --- |
| `1` | 当前用户已给该网站好评 |
| `2` | 当前用户已给该网站中评 |
| `3` | 当前用户已给该网站差评 |
| `null` | 当前用户未评价，或当前为游客 |

### 5.3 标签筛选在 ES 和 MySQL 路径下语义不同

查询参数 `tagNames` 在两条路径中真实行为不同：

#### ES 路径

- 使用 `termsQuery("tags", tagNames)`
- 语义更接近“命中任一标签即可”

#### MySQL 路径

- SQL 条件是 `t.tag_name IN (...)`
- 同样是“命中任一标签即可”
- 当前 `WebsiteQueryDTO` 里虽然有 `tagNameCount` 字段，但 mapper 并未使用 `HAVING` 做“多标签全匹配”

结论：

- 目前前后端都应按“标签或筛选”理解，而不是“必须同时命中全部标签”

### 5.4 网站提交的 URL 校验较简单

提交网站时，当前仅做了简单前缀校验：

- 必须以 `http://` 或 `https://` 开头

不会做更复杂的域名合法性校验、可访问性校验或重复 URL 校验。

### 5.5 网站提交有分布式锁

提交接口加了分布式锁注解：

- `scene = "website_submit"`
- `key = "website_submit_lock"`
- `includeUserId = true`

含义：

- 同一个用户提交网站时，短时间内会被串行化，防止重复提交

### 5.6 收藏接口的 `remark` 参数当前未落库

控制器 `POST /pc/website/favorite` 暴露了参数：

- `remark`

但当前 service 和 mapper 调用链里并没有把这个参数保存到 `osh_user_favorite_website` 表。

也就是说：

- 该参数目前对业务无实际效果

### 5.7 收藏是幂等恢复式设计

收藏表使用的是：

```sql
INSERT ... ON DUPLICATE KEY UPDATE delete_flag = 0
```

意味着：

1. 第一次收藏：插入记录
2. 取消收藏：`delete_flag = 1`
3. 再次收藏：不会重复插入，而是恢复原记录为有效状态

### 5.8 评价支持修改，不支持重复提交相同值

评价提交规则：

1. 未评价过：插入新评价
2. 已评价过且类型相同：报错 `您已经评价过了`
3. 已评价过且类型不同：允许修改评价

### 5.9 审核通过后的联动

管理员审核通过一个网站后，系统会联动执行：

1. 更新网站状态为 `1-已通过`
2. 写入公告栏一条记录
3. 将该网站同步到 ES

### 5.10 好评会写入动态栏

以下情况会写入动态栏：

1. 用户第一次评价且为好评
2. 用户修改评价后改成好评

动态文案格式：

- `张** 给「GitHub」点了好评`

### 5.11 首页热门网站并不是简单按点击数排序

首页热门网站使用了综合热度公式：

- 0.50 * 点击分
- 0.30 * 好评分
- 0.20 * 收藏分

再按创建时间倒序作为兜底排序。

### 5.12 模块前端路径存在一个特殊点

首页热门网站返回的 `detailUrl` 实际来自：

- `modulePathService.getListPath("usefull")`

也就是说当前返回的是“网站列表页路径”，不是某个网站详情页路径。

```json
        "goodCount": 123,
        "midCount": 12,

        "collectionCount": 35,

      }
    ],

    "pageNum": 1,
    "pageSize": 10
  }

```

#### 查询规则

ES 路径：

1. `websiteName` 在 `name` 和 `description` 上做全文检索
2. 标签用 `termsQuery(tags, tagNames)`
3. 有关键词时优先按相关性排序
4. 然后按 `ratingScore desc`

MySQL 降级路径：

1. 只查 `status = 1` 且 `delete_flag = 0`
2. `websiteName` 对应 `w.name LIKE`
3. `tagNames` 对应 `t.tag_name IN (...)`
4. 按 `rating_score desc`

#### 特别说明

1. 当前筛选只返回审核通过的网站
2. 带有效 token 时会回填 `myRatingType`
3. `tags` 返回的是逗号分隔字符串，不是数组

## 8. 用户交互接口

基础路径：`/pc/website`

### 8.1 提交网站

- 请求方式：`POST`
- 路径：`/pc/website/submit`
- 认证：需要登录
- 权限：`website:submit`

#### 请求体示例

```json
{
  "name": "Hugging Face",
  "url": "https://huggingface.co",
  "description": "AI 模型与数据集平台",
  "logoUrl": "https://huggingface.co/front/assets/huggingface_logo.svg",
  "tagNames": ["AI", "模型平台"]
}
```

#### 校验规则

1. `name` 和 `url` 不能为空
2. `url` 必须以 `http://` 或 `https://` 开头

#### 真实业务行为

1. 自动生成资源编号 `no`
2. 初始状态固定为 `0-待审核`
3. 初始点击数为 `0`
4. `createBy` 取当前登录用户名
5. 如果有标签：
   - 不存在的标签会自动创建
   - 建立网站与标签关系
   - 对应标签 `use_count + 1`
6. 会尝试发送提交通知邮件
7. 邮件发送失败不影响主流程

#### 成功响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": null
}
```

#### 常见失败场景

- `网站名称和链接不能为空`
- `网站链接格式不正确，请以 http:// 或 https:// 开头`
- `提交失败，请稍后重试`

### 8.2 收藏网站

- 请求方式：`POST`
- 路径：`/pc/website/favorite`
- 认证：需要登录
- 权限：`website:favorite`

#### 请求参数

当前接口不是 JSON Body，而是表单/Query 形式：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `websiteId` | Long | 是 | 网站 ID |
| `remark` | String | 否 | 收藏备注，但当前不会真正保存 |

#### 请求示例

```http
POST /pc/website/favorite?websiteId=1&remark=常用代码托管平台
token: xxxxx
```

#### 真实业务行为

1. 校验 `websiteId` 和当前用户 ID
2. 若已收藏且未取消，则直接报错
3. 网站 `collection_count + 1`
4. 收藏表使用 UPSERT 方式恢复历史软删除记录

#### 成功响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": null
}
```

#### 失败场景

- `网站 ID 和用户 ID 不能为空`
- `您已经收藏过该网站了`
- `收藏失败，请勿重复收藏`

#### 特别说明

- `remark` 参数当前不会落库，前端不要依赖

### 8.3 取消收藏网站

- 请求方式：`GET`
- 路径：`/pc/website/del`
- 认证：需要登录
- 权限：`website:favorite:cancel`

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |

### 8.5 提交网站评价

- 请求方式：`POST`
- 路径：`/pc/website/rating/submit`
- 认证：需要登录
- 权限：`website:rating:submit`
- 请求体：JSON

#### 请求体示例

```json
{
  "websiteId": 1,
  "ratingType": 1
}
```

#### 校验规则

1. `websiteId` 不能为空
2. `ratingType` 不能为空
3. `ratingType` 必须是 `1/2/3`
4. 网站必须存在且已审核通过

#### 真实业务行为

##### 第一次评价

1. 新增评价记录
2. 对应网站的 `good_count / mid_count / bad_count` 递增
3. 如果是好评或差评，更新 `rating_score`
4. 如果是好评，写入动态栏
5. 同步最新评价计数到 ES

##### 修改评价

1. 如果新旧类型相同，直接报错 `您已经评价过了`
2. 如果不同，则更新评价类型
3. 对应新旧计数做增减平衡
4. 如果改成好评或差评，更新 `rating_score`
5. 如果改成好评，写入动态栏
6. 同步最新评价计数到 ES

#### 成功响应

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": null
}
```

#### 常见失败场景

- `请先登录`
- `参数不能为空`
- `评价类型必须是1(好评)、2(中评)或3(差评)`
- `网站不存在或未通过审核`
- `您已经评价过了`
- `评价失败，请稍后重试`

## 9. 管理审核接口

基础路径：`/pc/website`

### 9.1 审核网站

- 请求方式：`POST`
- 路径：`/pc/website/audit`
- 认证：需要登录
- 权限：`website:audit`
- 请求体：JSON

#### 请求体示例：通过

```json
{
  "websiteId": 20,
  "status": 1
}
```

#### 请求体示例：拒绝

```json
{
  "websiteId": 20,
  "status": 2,
  "rejectReason": "网站内容不符合规范"
}
```

#### 审核规则

1. `websiteId` 不能为空
2. `status` 只能是 `1-通过` 或 `2-拒绝`
3. 拒绝时 `rejectReason` 必填
4. 当前实现查询的网站必须满足：
   - `delete_flag = 0`
   - `status = 0`

也就是说：

- 只能审核待审核网站
- 已审核网站再次调用通常会返回“网站不存在”

#### 通过时联动

1. 更新网站状态
2. `auditBy = "admin"`
3. `auditTime = 当前时间`
4. 写入公告栏
5. 将网站同步到 ES

#### 拒绝时联动

1. 更新状态为 `2-已拒绝`
2. 写入 `rejectReason`

#### 返回

通过：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": "审核通过"
}
```

拒绝：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": "已拒绝"
}
```

#### 失败场景

- `网站 ID 不能为空`
- `审核状态必须是 1（通过）或 2（拒绝）`
- `拒绝时必须填写拒绝原因`
- `网站不存在`

## 12. 典型前端接入流程

### 12.1 游客浏览流程

1. `GET /pc/website/tags` 获取标签候选
2. `POST /pc/website/list` 获取网站列表
3. `PUT /pc/website/click?id=xxx` 记录点击
4. `GET /pc/website/notices` 获取公告栏
5. `GET /pc/website/dynamics` 获取动态栏
6. `GET /pc/homepage/website/hot` 获取首页热门网站

### 12.2 登录用户交互流程

1. 登录后调用 `POST /pc/website/list`
2. 列表中会额外得到 `myRatingType`
3. 收藏网站：`POST /pc/website/favorite`
4. 查看收藏：`GET /pc/website/Favorites`
5. 取消收藏：`GET /pc/website/del`
6. 评价网站：`POST /pc/website/rating/submit`

### 12.3 审核流程

1. 用户提交网站：`POST /pc/website/submit`
2. 管理端查看待审核列表：`GET /pc/website/audit/list`
3. 查看待审核详情：`GET /pc/website/audit/detail/{websiteId}`
4. 审核通过或拒绝：`POST /pc/website/audit`
5. 审核通过后：
   - 公告栏新增
   - ES 同步

## 13. 代码层注意事项

### 13.1 列表接口使用 `POST`，不是 `GET`

公开网站列表接口当前是：

- `POST /pc/website/list`

不是常见的 GET 查询接口。

### 13.2 收藏接口不是 JSON Body

`POST /pc/website/favorite` 当前参数绑定方式更像表单/Query 参数，不是 `@RequestBody`。

### 13.3 取消收藏和批量删除都用了 `GET`

当前存在两个“不太 RESTful”的接口：

- `GET /pc/website/del`
- `GET /pc/website/batch`

前端联调时需要严格按当前代码调用。

### 13.4 我的收藏路径 `Favorites` 大写

接口路径是：

- `/pc/website/Favorites`

不是 `/favorites`。

### 13.5 收藏备注参数当前无效

控制器有 `remark` 参数，但当前不会保存到数据库。

### 13.6 批量删除后未同步清理 ES

当前删除网站主表和标签关联表后，没有同步删除 ES 文档，因此如果 ES 中仍有旧数据，可能要靠重建索引或全量同步修正。

### 13.7 批量删除后未回滚标签使用次数

网站标签 `use_count` 在绑定时会增加，但批量删除网站时当前没有减少对应标签的使用次数。

### 13.8 审核人当前写死为 `admin`

审核接口在 service 中设置：

- `auditBy = "admin"`

当前不是从真实登录管理员上下文动态取值。
